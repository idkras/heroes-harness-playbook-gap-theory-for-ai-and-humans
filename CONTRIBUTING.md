# Contributing to Heroes Gap Theory

Спасибо за интерес. Этот репо — методологическая база, и контрибьюции через PR / Issues / Discussions приветствуются.

## Что особенно нужно

| Тип контрибьюции | Приоритет | Куда |
|---|---|---|
| Worked examples из не-ecommerce доменов (психология, образование, государственное управление, медицина, hardware) | 🔥 High | `worked-examples/` или `examples/` |
| Перевод стандартов и скилов на английский | 🔥 High | issue first, PR per file |
| Уточнение формул Glue Effort | High | `docs/glue-effort-formulas.md` (создать) |
| Bug fixes в текстах (опечатки, broken links, formatting) | Medium | прямой PR |
| Новые скилы — production-tested, не теоретические | Medium | issue first для согласования с Gap Theory canon |
| Расширение шаблонов | Medium | `templates/` |
| Application notes к существующим стандартам | Medium | `docs/applications/` (создать) |

## Workflow

### 1. Issue — для всего нового и спорного

Перед PR с большим изменением (новый скилл, новая теория, переписка стандарта) **открой Issue** с:

- Что предлагаешь
- Картина мира (твоя текущая) + объяснительная модель (как это связано с существующим)
- Что уже пробовал / какие альтернативы рассмотрел
- Один worked example «как бы это работало» (1 страница, реальная ситуация)

Это сэкономит и тебе и мне время на review.

### 2. PR — для конкретных правок

- Один PR = одна целевая правка (один файл правится — отдельный PR; один новый скилл — отдельный PR)
- Заголовок PR по формату `feat: <что добавлено>` / `fix: <что починено>` / `docs: <что улучшено>` / `translate: <что переведено>`
- В описании PR — обязательно: ссылки на связанные стандарты, какие скилы затрагиваются, чек-лист «что я проверил»
- Если правишь стандарт — добавь changelog entry в `CHANGELOG.md` под `[Unreleased]`

### 3. Discussions — для применения и кейсов

Если ты применил Gap Theory к чему-то новому и хочешь рассказать (но это не готово как worked-example) — открой Discussion. Я переношу удачные обсуждения в `examples/` через PR от автора.

## Стандарт качества для новых worked-examples

Worked example засчитывается если:

1. **Реальный контекст** (или собирательный, но не выдуманный) — есть конкретные цифры, цитаты, действия
2. **Анонимизация** — реальные имена/домены/цифры заменены, но subject area сохранена
3. **4 типа гепов размечены** (knowledge / execution / feedback / integration)
4. **Glue Effort оценён** по шкале 0-100
5. **Гипотеза и фальсификация** — есть expectation-vs-reality таблица
6. **Verdict** — что сработало, что нет
7. **Длина** 1-3 страницы. Длиннее — скорее всего над-универсализирован.

## Что не пройдёт review

- Тексты без worked examples («теоретическое расширение без применения»)
- Универсализация под все случаи без описания границ применимости
- Привязка к одному вендору (например, «Gap Theory для пользователей конкретного CRM») — должно быть переносимо
- Конфликт с существующими стандартами без явного rationale «почему старое неверно»
- Прямой перевод без культурной адаптации

## Code of conduct

Простое: relentlessly constructive.

- Критика — на содержание, не на автора
- Альтернативная гипотеза приветствуется, но **с** объяснительной моделью почему
- Если не согласен — покажи где gap в моей модели мира, не «не нравится»
