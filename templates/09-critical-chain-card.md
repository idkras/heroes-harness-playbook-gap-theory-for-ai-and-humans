# Template: Critical Chain Card (Goldratt-style)

> Шаблон проектирования критической цепочки задач от Голдрата. 5 шагов: Outcome → ограничение → цепочка от конца → проверка длины → buffer.

## Принцип Голдрата

Любой проект ограничен **одной критической последовательностью** — самой длинной цепочкой зависимостей от старта до выхода. Любые другие задачи — параллельные ветки. Оптимизировать надо **только критическую цепочку**, остальное — buffer.

## Шаблон

### 1. Outcome (что должно случиться в реальности)
```
Real outcome: <одна фраза, измеримая>
Например: «продукт активирован у 50 платящих клиентов через 12 недель»
```

### 2. Ограничение (что узкое горло)
```
Constraint: <одна фраза>
Например: «один человек продаёт = capacity 5 продаж в неделю»
```

### 3. Цепочка от конца к началу (3-5 шагов)

```
Step N (outcome): <real outcome>
       ↑
Step N-1: <что нужно непосредственно перед outcome>
       ↑
Step N-2: <что нужно перед N-1>
       ↑
Step N-3: <что нужно перед N-2>
       ↑
Step 1 (start): <ближайший action который можно сделать сегодня>
```

**Длина цепочки:** 3-5 шагов. Если получилось 7+ — слишком многое в критической цепочке, выноси параллельные ветки.

### 4. Параллельные ветки (что НЕ в критической цепочке)

```
Parallel branch A: <задача> — собирается параллельно к step N-2
Parallel branch B: <задача> — собирается параллельно к step N-3
```

### 5. Buffer (где запас времени)

```
- Buffer перед outcome: X недель — для интеграции последних шагов
- Buffer перед каждым параллельным merge: Y дней
- Total buffer: 30% от общей длительности
```

## Worked example: запуск новой версии продукта за 12 недель

```
Outcome: продукт версии v2.0 активирован у 50 платящих клиентов
Constraint: один solution architect для онбординга (capacity 5 онбордингов/неделю)

Critical chain (от конца):

Step 5 (outcome): 50 активированных платящих клиентов
       ↑ buffer 2 недели
Step 4: 80 клиентов прошли онбординг (assumption: 60% retention)
       ↑ buffer 1 неделя
Step 3: 80 клиентов согласились на pilot v2.0
       ↑ buffer 0 — параллельный pipeline продаж
Step 2: v2.0 готова к onboard (no critical bugs)
       ↑ buffer 1 неделя — final QA
Step 1 (start): команда начала разработку v2.0

Parallel branches:
A. Команда продаж формирует pipeline на 100 кандидатов — параллельно step 1-3
B. Документация и материалы онбординга — параллельно step 2

Buffer: 4 недели итого (33% от 12 недель)
```

## Шаги Голдрата (5 focusing steps)

1. **Identify** the constraint (узкое горло)
2. **Exploit** the constraint (выжать из него максимум без расширения)
3. **Subordinate** everything else to the constraint (всё остальное должно служить узкому горлу)
4. **Elevate** the constraint (расширить узкое горло если 1-3 не достаточно)
5. **Repeat** — после устранения constraint новый узким горлом станет другое

## Anti-patterns

- ❌ Цепочка длиннее 7 шагов — слишком линейно, есть пропущенные параллельные
- ❌ Все шаги критические — значит ничего не критическое
- ❌ Buffer = 0 — гарантированный slip
- ❌ Outcome = «улучшить» — не измеримо, не подходит
- ❌ Constraint = «не хватает времени» — не constraint, а отсутствие приоритизации
