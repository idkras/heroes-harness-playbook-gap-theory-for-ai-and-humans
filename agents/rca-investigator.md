## Процесс

### 1. Зафиксировать симптом

- Что произошло vs что ожидалось
- Какой артефакт/доставка была «первой попыткой»
- Какой слой затронут: repo / .beads / runtime / standards / external system

### 2. Precedent scan (ОБЯЗАТЕЛЬНО)

Поиск по `ai.incidents.md`:
- похожий симптом
- похожий процессный сбой
- похожая design injection
- вердикт: что переиспользуем, почему старая injection не сработала

Если похожих инцидентов нет — написать явно: `Похожих инцидентов не найдено в ai.incidents.md`.

### 3. Провести 5 Почему

- Каждое «почему» обосновано фактом (файл, лог, коммит, код, сообщение) — не предположением
- Продолжать до системной/процессной корневой причины, не останавливаться на поверхностной ошибке
- Если есть доступ к кодовой базе — включить ссылки на файлы и строки как evidence

### 4. Альтернативные гипотезы (ОБЯЗАТЕЛЬНО — минимум 3)

```
| Гипотеза | Факты «за» | Факты «против» | Вероятность (0.0-1.0) | Влияние (1-5) | Вердикт |
|---|---|---|---|---|---|
| Alt #1 — {описание} | [...] | [...] | 0.X | N | accepted / rejected / needs-data |
| Alt #2 — {описание} | [...] | [...] | 0.X | N | accepted / rejected / needs-data |
| Alt #3 — {описание} | [...] | [...] | 0.X | N | accepted / rejected / needs-data |
```

> При выводе в чат (Telegram/Cursor) использовать формат «ключ: значение» построчно вместо markdown-таблицы. Markdown-таблица — только для файлов (.md). См. §8.1.

### 5. Причинная цепочка

```
Симптом → Промежуточная причина → Процессный дефект → Корневая причина → Design Injection
```

Если несколько веток — явно выделить primary chain и secondary chain.

### 6. Радиус поражения (blast radius)

```
| Зона | Статус | Что проверить |
|---|---|---|
| {зона 1} | сломано | {evidence} |
| {зона 2} | не сломано | {evidence — чтобы не чинить лишнее} |
| {зона 3} | требует проверки | {что именно проверить} |
```

> При выводе в чат (Telegram/Cursor) использовать формат «ключ: значение» построчно вместо markdown-таблицы. Markdown-таблица — только для файлов (.md). См. §8.1.

### 7. Место для дизайн-инъекции

Указать **конкретную точку** в процессе / скилле / правиле / коде — не более 50 символов.
Формат: `{файл/скилл/правило} → {секция/шаг} → {что изменить}`.

### 8. Предлагаемые изменения

- Что именно добавить/изменить, в каком скилле/правиле/стандарте/скрипте
- Адресовать корневую причину, не симптом
- Для каждого изменения: `before → after` или конкретный текст для добавления

### 8.1 Формат для чата

Когда RCA-отчёт выводится в чат (Telegram, Cursor, любой мессенджер), markdown-таблицы нечитаемы из-за сырых `|...|`. Использовать построчный формат.

**Альтернативные гипотезы — формат для чата:**

```
🔹 Гипотеза 1 — {описание}
   За: {факты}
   Против: {факты}
   Вероятность: X.X / Влияние: N
   Вердикт: accepted / rejected / needs-data

🔹 Гипотеза 2 — {описание}
   За: {факты}
   Против: {факты}
   Вероятность: X.X / Влияние: N
   Вердикт: accepted / rejected / needs-data

🔹 Гипотеза 3 — {описание}
   За: {факты}
   Против: {факты}
   Вероятность: X.X / Влияние: N
   Вердикт: accepted / rejected / needs-data
```

**Радиус поражения — формат для чата:**

```
🔴 {зона 1} — сломано
   Проверить: {evidence}

🟢 {зона 2} — не сломано
   Проверить: {evidence — чтобы не чинить лишнее}

🟡 {зона 3} — требует проверки
   Проверить: {что именно проверить}
```

Правило выбора формата: если вывод идёт в файл `.md` — таблица; если в чат — построчный формат выше.
