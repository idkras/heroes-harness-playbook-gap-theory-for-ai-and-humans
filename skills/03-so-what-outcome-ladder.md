# So-What Outcome Ladder — отсечение псевдо-outcomes

## Назначение

Когда агент завершает работу и формулирует результат, часто то, что выглядит как outcome, на самом деле — промежуточный checkpoint. Этот скилл нанимается на работу **проверить каждый заявленный outcome через фильтр «И что с того?»** и вывести только настоящие результаты на поверхность.

## Hired for JTBD — задача, которую решает этот скилл

Когда агент заявляет «задача готова» с набором результатов, отличить настоящие outcomes (меняют поведение оператора/пользователя/процесса) от псевдо-outcomes (промежуточные шаги, которые не имеют самостоятельной ценности).

## When to use

- После фиксации A → G в task-orchestrator — проверить, что G не псевдо
- Перед закрытием bead — проверить, что DoD-пункты = настоящие outcomes
- Когда пользователь спрашивает «и что с того?», «это настоящий результат?»
- В protocol-challenge перед вынесением verdict
- При планировании через `1-next` — проверить, что outcome = outcome, а не checkpoint

## Input checklist

- [ ] Есть список заявленных outcomes / DoD-пунктов / результатов
- [ ] Понятно, кто следующий оператор или пользователь (чьё поведение должно измениться)
- [ ] Понятно, что было **до** (точка A) и что заявлено **после** (точка G)

## Количественная метрика outcome — gap score

Для формального сравнения двух outcome между собой («какой ближе к JTBD») применяется gap score по стандарту 1.15 §7-8:

```
gap = (knowledge gap + execution gap + feedback gap + integration gap) × 100 / 12
```

Шкала каждого измерения 0-3 (Standard 1.15 §6), итог 0-100. Чем ближе к 0 — тем outcome точнее закрывает JTBD. Параллельная метрика `effort score` (0-100) считает сколько ручного glue остаётся у человека.

**Пороги запуска:**
- goodness = 100 − √(gap × effort) ≥ 70 → pass (задача готова)
- 50 ≤ goodness < 70 → rework
- goodness < 50 → stop (псевдо-outcome)

**CLI:** `python3 scripts/gap_effort_calculator.py --knowledge-gap 0-3 --execution-gap 0-3 --feedback-gap 0-3 --integration-gap 0-3 --cognitive-effort 0-3 --mechanical-effort 0-3 --integration-effort 0-3 --coordination-effort 0-3`

**Калибровка:** `docs/gap-effort-calibration.md` (5 полос статусов + 3 разобранных примера).

Применяется outcome-designer субагентом автоматически. Для ручного использования — вызвать CLI после формулировки outcome.

## Core route

### 0. Макрос §0 «ID + человекочитаемое имя» (ОБЯЗАТЕЛЬНО)

Все ID (G01, P0, E01, pr-rick-N) — только с подписью через ` — `. Канон: `AGENTS.md` §0 и `.agents/skills/2-hypothesis-gap-falsification/SKILL.md` §0.

### 1. Собрать список заявленных outcomes

Из bead DoD, из A → G фиксации, из output checklist — собрать всё, что заявлено как результат.

### 2. Прогнать каждый через фильтр «И что с того?»

Для каждого заявленного outcome задать вопрос: **«И что с того для следующего оператора / пользователя?»**

Три уровня ответа:

| Уровень | Тип | Пример | Verdict |
| --- | --- | --- | --- |
| **L0 — Действие** | Файл создан, команда выполнена, bead обновлён | «Snapshot JSON создан» | ❌ Не outcome — это действие |
| **L1 — Checkpoint** | Проверка пройдена, страница открывается, тест зелёный | «Страница открывается в браузере» | ⚠️ Checkpoint, не outcome — нужен ещё один «и что?» |
| **L2 — Outcome** | Поведение изменилось, процесс стал короче/безопаснее, ручной шаг устранён | «Оператор больше не восстанавливает вручную связь sheet → hierarchy → parquet → UI» | ✅ Настоящий outcome |

### 3. Для каждого псевдо-outcome — найти настоящий

Если заявленный outcome = L0 или L1, задать ещё один «и что с того?» и вывести на L2.

Формат:

```markdown
### So-What Ladder

| # | Заявленный outcome | Уровень | И что с того? | Настоящий outcome (L2) |
|---|-------------------|---------|---------------|----------------------|
| 1 | Snapshot JSON создан | L0 — действие | Данные зафиксированы → и что? | Следующий оператор видит актуальную иерархию без ручного чтения sheet |
| 2 | Страница открывается | L1 — checkpoint | Визуально работает → и что? | Оператор может показать клиенту живой виджет без сборки |
| 3 | Bead обновлён | L0 — действие | Статус актуален → и что? | Следующий агент начинает с правильной точки A, не тратит время на recovery |
```

### 4. Вынести итоговый outcome

Из всех L2-outcomes собрать **один главный outcome** задачи — тот, ради которого всё делалось:

```markdown
### Итоговый outcome
{Одно предложение: что изменилось для следующего оператора / пользователя / процесса}
```

### 5. Проверить: G из task-orchestrator = итоговый outcome?

Если G (целевая точка) задачи не совпадает с итоговым outcome — это gap. Зафиксировать:

```markdown
### Gap: G ≠ итоговый outcome
- **G заявленное:** {что было в A → G}
- **Итоговый outcome:** {что получилось по so-what ladder}
- **Δ:** {в чём расхождение}
```

## Типичные псевдо-outcomes (антипаттерны)

- «Файл создан» → L0: действие, не outcome
- «Тест прошёл» → L1: checkpoint
- «Bead обновлён» → L0: действие
- «Sheet прочитан» → L0: действие (это вход, не результат)
- «PR создан» → L1: checkpoint (PR ≠ доставка)
- «Страница открывается» → L1: checkpoint
- «Данные выгружены» → L0: действие

## Типичные настоящие outcomes

- «Оператор больше не делает ручной шаг X»
- «Цикл интеграции с Y стал короче на Z шагов»
- «Следующий агент начинает с подтверждённой точки A, а не с recovery»
- «Клиент видит живую поверхность без ожидания сборки»
- «Процесс N стал воспроизводимым: любой оператор может повторить по DoD»

## Forbidden

- Принимать L0/L1 за outcome без прохождения фильтра «и что с того?»
- Остановиться на одном «и что?» — нужно дойти до L2
- Подменять outcome действием: «мы сделали X» ≠ «X изменил Y»
- Пропускать проверку G ≠ итоговый outcome

## Outcome target

- Все заявленные outcomes прошли через so-what фильтр
- Псевдо-outcomes (L0, L1) отделены от настоящих (L2)
- Итоговый outcome сформулирован в одно предложение
- G из task-orchestrator проверена на соответствие итоговому outcome

## Язык результата

Весь человекочитаемый результат — на русском. Английский допустим только для точных имён API, методов, идентификаторов кода и меток вендорских интерфейсов.

## Reasoning Log Protocol (ОБЯЗАТЕЛЬНО)

При каждом исполнении этого скилла агент ОБЯЗАН вести reasoning log в чате (формат — см. `AGENTS.md` Reasoning Log Protocol). §0 macro: все ID с подписью. Hard fail без reasoning log.

## Owner value

Каждое исполнение скилла приносит owner value: отделение настоящих результатов от промежуточных шагов, ускорение переговоров с заказчиком (вместо «мы сделали X» → «благодаря X следующий оператор экономит Y минут»), улучшение quality гейта перед закрытием задач. Value_per_touch: один so-what run = одна проверенная итоговая ценность = confidence что G реально достигнут.

## Связанные скилы

- `1-task-orchestrator` — мета-оркестратор, вызывает so-what после A → G
- `1-next` — планирование, где outcome должен быть проверен через so-what
- `2-protocol-challenge` — финальная фальсификация, где so-what применяется к выводам
- `2-hypothesis-gap-falsification` — проверка гипотез, где outcome ≠ checkpoint
