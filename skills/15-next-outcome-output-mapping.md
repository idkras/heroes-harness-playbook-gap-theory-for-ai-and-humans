# Next — план, outcome и инкрементальный цикл (HADI)

## Назначение

**JTBD:** Когда нужно понять «что делаем дальше по проекту», выписать output / outcome релиза, проверить, получишь ли ты пользу, и при необходимости скорректировать план через **критическую цепочку** и **разницу outcome ожидаемый − outcome фактический** — использовать этот скилл.

Скилл структурирует работу через:
- **Situation/Problem (SGR):** в чате явно фиксируется «ситуация-проблема» до плана и решений
- **Input checklist** — что нужно на входе (проект, todo, последние вводные)
- **Output checklist** — что создаётся/доставляется
- **Outcome checklist** — какая выгода/польза для тебя, как проверить
- **From-the-end chain:** сначала `outcome (какая польза и что изменится в процессе людей/команды) -> output (какой артефакт это докажет) -> blockers/constraints -> минимальная критическая цепочка`
- **HADI-цикл** (Hypothesis → Action → Data → Insights): план → уточнение по новым вводным → доделать → запустить код → проверить гипотезу → скорректировать план по реальным данным (outcome ожидаемый − outcome фактический)

**Based on:** [core-auto.mdc](.cursor/rules/core-auto.mdc) (OUTPUT / OUTCOME TRACKING, ITERATIVE DEVELOPMENT, REFLECTION CHECKPOINTS), [core-next.mdc](.cursor/rules/core-next.mdc) (план → auto → check → clean → next actions), [From-The-End 1.4](<standard-ref>) (критическая цепочка = последовательность зависимых задач, минимизация простоя), [Standard 5.30](<standard-ref>) (п. 12.4 минимальная критическая цепочка DBR, Drum-Buffer-Rope — барабан-буфер-канат по Голдрату), [SGR 2.9](<standard-ref>) (schema-guided шаги к outcome).

## When to Use

- «Что делаем дальше по проекту?»
- «Выпиши output/outcome релиза, проверь outcome»
- «Нужно изменить план релиза через критическую цепочку»
- В начале сессии: выписать уточнённый план с учётом изменений в чате и проверки гипотез на реальных данных
- После итерации: что доделать, чтобы ты получил пользу и output

## Prompt pack without naming the skill

Use these phrasings when you want the same behavior but do not want to mention internal skill names:

- `Разложи задачу от конечного результата назад: какой output нужен, какой outcome докажет пользу и что является первой критической цепочкой.`
- `Покажи, что нужно сделать дальше по проекту, чтобы получить первый измеримый результат, а не просто список задач.`

- `Сравни ожидаемый outcome и фактический outcome, покажи gap и как его закрыть следующим циклом.`
- `Сделай план не от действий, а от итогового результата для пользователя.`

## Invocation — обязательное выполнение

**Когда пользователь пишет «skill next», «скилл next», «давай next» (или эквивалент)** — агент **в этой же сессии** обязан выполнить полный протокол, а не использовать скилл только как справочник.

**Обязательный порядок в этой сессии:**

1. **Прочитать контекст** — открыть `{projectname}.todo.md` (из чата или последнего упомянутого проекта), секции OUTPUT/OUTCOME, БЛОКЕРЫ, СЛЕДУЮЩИЕ ШАГИ.
2. **Если задача затрагивает БД, API или другой источник данных:** сначала выполнить [DATA SOURCES — RESEARCH-FIRST](#data-sources--research-first) — прочитать источник, изучить структуру (схема, типы, head, uniques), выписать в чат; только после этого формулировать vision/план и переходить к шагу 3. Иначе — cherry-picking и survivorship bias (см. core-auto.mdc и research.mdc).
3. **Выписать в чат SGR-каркас + OUTPUT/OUTCOME** — обязательно: `Ситуация-проблема`, `JTBD`, `Input checklist`, `Output checklist`, `Outcome checklist`, `BLOCKERS`, `NEXT STEPS`; outcome фактический (если уже что-то делали); gap (ожидаемый − фактический); критическая цепочка при необходимости.
   - Порядок обязателен: `Outcome/JTBD change -> Output artifact -> Blockers -> Critical chain`.
   - Нельзя начинать с полного списка подзадач. Сначала задача должна быть сведена к 1-3 шагам критической цепочки, которые дают первый usable outcome.
4. **Проверить: получишь ли пользу** — если outcome фактический ещё нет — написать, что сделать и запустить для первого измеримого outcome; если gap не нулевой — предложить корректировку плана.
5. **Уточнённый план** — с учётом изменений в чате и гипотез; что доделать, в каком порядке; что проверить на реальных данных.
6. **При необходимости — изменения и проверка** — сделать изменения (один шаг за раз), тут же проверить (запуск, тесты, данные); не переходить к следующему шагу без проверки. **Если блокер = «применить миграцию» / «запустить скрипт X»** — в этой сессии **сначала запустить** этот скрипт (env/Keychain); только при ошибке (нет кредов) — сообщить и просить добавить; не писать «нужно сделать вручную» без попытки запуска (ai.incidents 15 Feb 2026).
7. **Доставка результата и польза** — в конце явно написать: что доставлено (артефакты, ссылки, статус); **какие изменения реально применимы сейчас** (команды/файлы/настройки); **«Польза тебе: да / нет»** — получил ли пользователь ценность/результат по факту (с кратким обоснованием).
8. **Уверенность и калибровка** — для каждого шага/результата писать в чат уверенность по шаблону: `Confidence: [до]% → [после]% — [причина]`. Использовать [Confidence Calibration Matrix](.cursor/rules/core-check.mdc#2-confidence-calibration-matrix) из core-check.mdc (например: код не запущен = -50%, данные не проверены = -55%). Завершать без указания уверенности запрещено.
9. **Trust (доверие к результату)** — в конце выписать Trust по [trust-metric](.agents/skills/2-trust-metric/SKILL.md): `📊 TRUST (доверие к результату): [X] / 100`; 100 = минимальный steering, нет инцидентов; каждое направление/коррекция и инцидент снижают Trust. Явно писать в чат.
10. **Метрики процесса (обязательный блок):** incidents, gaps, steering, trust, confidence. Без метрик завершать ответ нельзя.

**Запрещено при вызове «skill next»:** ограничиваться только правками в документах или частичным выполнением; завершать без явной формулировки «доставка» и «польза тебе: да/нет»; при блокере «запустить скрипт» — только описать план без запуска скрипта в сессии.

*См. инцидент 15 Feb 2026 в ai.incidents.md (полный протокол next не выполнен при вызове по имени); Standard 1.13 — типовой гэп «Вызов скилла по имени».*

## Data sources — RESEARCH-FIRST {#data-sources--research-first}

**Когда задача затрагивает БД, API или другой источник данных** (Supabase, Postgres, Parquet, JSON/CSV, внешний API):

- [ ] **Сначала:** прочитать источник и изучить структуру (схема, колонки/типы, head(5), uniques по ключевым полям). Следовать [research.mdc](.cursor/rules/research.mdc) (DATA RESEARCHER PROTOCOL). Выписать в чат пути, имена таблиц, список полей и примеры значений.
- [ ] **Только после этого:** формулировать vision, OUTPUT/OUTCOME и план; затем реализовывать и тестировать.

**Запрещено:** составлять детальный план или писать код, читающий/пишущий источник, **до** того как загружена и показана реальная структура. Иначе — cherry-picking и survivorship bias. **Design injection — RCA, Root Cause Analysis:** чекпоинт «сначала данные, потом план и код» устраняет режим «сначала план/код, потом глянем данные».

## Input checklist

Перед выписыванием плана и «next» убедиться:

- [ ] **Проект и todo:** С каким `{projectname}.todo.md` работаем? Путь к файлу указан.
- [ ] **Текущий релиз/блок:** Какой релиз или блок задач в фокусе (из todo или из чата)?
- [ ] **Источники данных (если есть):** Если в задаче есть БД / API / файлы — выполнено ли RESEARCH-FIRST (структура прочитана и выписана в чат)? См. [Data sources — RESEARCH-FIRST](#data-sources--research-first).
- [ ] **Последние вводные из чата:** Что пользователь изменил, уточнил, что «потеряли» с прошлых действий?
- [ ] **Гипотезы и проверки:** Какие гипотезы уже проверялись на реальных данных? Результат (подтверждено/опровергнуто)?
- [ ] **Блокеры:** Какие блокеры из todo ещё не сняты? Критическая цепочка (Голдрат): какая цепь зависимых задач до первого измеримого outcome и где минимум зависимостей?

---

## Output checklist (что создаётся/доставляется в этом релизе)

- [ ] **Артефакты:** Конкретные файлы, скрипты, документы, ссылки. Не «обновления», а пути и имена.
- [ ] **Формат/структура:** В каком виде выход (JSON, таблица, скрипт с --profile, и т.д.).
- [ ] **Критерий «готово»:** При каком условии output считается доставленным (тест прошёл, миграция применена, скрипт запущен с --profile lisa и т.д.).
- [ ] **Critical-chain relevance:** Каждый output реально нужен для первого outcome, а не является боковой активностью или inventory.

---

## Outcome checklist (польза/выгода для тебя)

- [ ] **Outcome ожидаемый:** Что ты получишь и сможешь делать после этого релиза (одним предложением на ценность).
- [ ] **Качественная гипотеза изменения:** Что изменится в процессе команды/людей/JTBD-сценарии после работы, даже если метрики пока не посчитаны.
- [ ] **Outcome фактический (если уже что-то делали):** Что уже получилось по факту (по логам, запускам, данным).
- [ ] **Gap:** outcome ожидаемый − outcome фактический. Что не хватает, чтобы получить пользу?
- [ ] **Критическая цепочка (по Голдрату):** Цепь зависимых задач проекта, определяющая минимальное время до outcome (см. From-The-End 1.4, Standard 5.30 п. 12.4). Не путать с DBR, Drum-Buffer-Rope — барабан-буфер-канат — это про производство; критическая цепочка — про проект и порядок задач с минимумом блокеров.
- [ ] **Нужно ли менять план:** Если gap большой — скорректировать план (следующие шаги) так, чтобы следующий цикл закрывал gap и ты получил измеримый outcome.

## SGR Offer Chat Format (MANDATORY)

При `next` ответ в чат должен содержать этот каркас:

```text
🧩 СИТУАЦИЯ-ПРОБЛЕМА
- Ситуация: ...
- Что не работает: ...
- Почему это важно сейчас: ...

🎯 JOBS TO BE DONE, JTBD — КОТОРЫЙ РЕШАЕМ
- Когда ... я хочу ... чтобы ...
- Success metric: ...

📥 INPUT CHECKLIST
- [ ] ...

📦 OUTPUT CHECKLIST
- [ ] ...

✅ OUTCOME CHECKLIST
- [ ] Ожидаемая выгода: ...
- [ ] Фактическая выгода: ...
- [ ] Польза тебе: Да / Нет

🛠️ РЕАЛЬНО ПРИМЕНИМЫЕ ИЗМЕНЕНИЯ
1. Команда/изменение ...
2. Команда/изменение ...

📊 PROCESS METRICS
- Incidents (new): N
- Gaps (open): N
- Steering events: N
- Trust: X/100
- Confidence: A% → B%
```

Правила:
- `Incidents` = только новые инциденты текущего цикла (если 0, писать `0`).
- `Gaps` = число незакрытых разниц «outcome ожидаемый − outcome фактический».
- `Steering events` = количество пользовательских корректировок направления в текущем цикле.
- `Польза тебе: Да` ставить только если outcome подтверждён фактом (запуск/данные/лог), иначе `Нет`.

## Unified Delivery Format (MANDATORY)

В финальном ответе по `next` всегда включать:

1. **Было/Стало**
- Было: что отсутствовало или где был gap.
- Стало: что конкретно изменено (файлы, протокол, проверки, команды).

2. **JTBD, Jobs To Be Done — задача, которую решает клиент - сценарий (коротко)**
- Когда, Роль, Хочет, Закрывает потребность, Делает, Мы хотим.

3. **Input / Output / Outcome Checklists**
- Явно показать прогон по чеклистам (`[x] / [ ]`) с фактическими пунктами текущей задачи.

4. **Run Evidence**
- Команды, которые реально запускались.
- Что фактически получено (PASS/FAIL, ключевые ошибки/дельты).
- Где лежат артефакты/логи.
- Для data-critical: явно указать `A-B` и `B-A` (commutator), если применимо.

**FORBIDDEN:** финальный ответ без блока `Было/Стало` и без `Run Evidence`.
**FORBIDDEN:** отвечать на `что делаем дальше`, `next steps`, `status`, `project summary` только prose-планом, если задача packet-like или multi-step.

## Ticket-card output contract

Для `next` human-facing output по умолчанию должен рендериться как `1-3 Quick ticket cards`, а не как свободный план.

Минимум:

1. card для текущего epic / project slice
2. card для следующего executable slice
3. card для blocker / missing delivered slice, если он определяет критическую цепочку

Каждая card обязана начинаться не с подзадач, а с цепочки:

1. `Outcome / какая польза и какое изменение процесса хотим`
2. `Output / какой артефакт это докажет`
3. `Blocker / constraint / unknown`
4. `Critical-chain subtask`

### Mandatory hypothesis block in every card

В каждой ticket card обязан быть отдельный блок:

```md
Гипотеза изменения:
- после этого релиза изменится:
- в процессе команды / людей станет:
- в JTBD-сценарии станет проще / безопаснее / быстрее:
- больше не нужно будет вручную:
```

### Subtask reduction rule

- `Sub-tasks` в card должны показывать только критическую цепочку до первого outcome.
- Если подзадача не снимает текущий constraint, не даёт первый usable output и не нужна для проверки гипотезы, её нельзя включать в card первого релиза.
- Если получается 4+ равновесных подзадачи, значит цепочка не найдена и план нужно пересобрать с конца.

## HADI-цикл (инкрементальный релиз)

Цикл из [core-auto](.cursor/rules/core-auto.mdc) (итеративная разработка, один шаг — тест — отражение) в явном виде:

| Фаза | Действие | Чеклист для агента |
|------|----------|---------------------|
| **H**ypothesis | Выписать план и гипотезу | Что мы проверяем в этой итерации? Какой outcome ожидаем? |
| **A**ction | Доделать что нужно, запустить код | Одно изменение → тест. Не несколько правок без проверки. |
| **D**ata | Собрать факты (логи, вывод скрипта, ответ API) | Что реально произошло? Цифры, статусы, ошибки. |
| **I**nsights | Сравнить ожидаемое и фактическое | outcome ожидаемый − outcome фактический = gap. Корректировка плана на основе gap. |

После **Insights** — обновить план (next steps) в `{projectname}.todo.md` и в чате: что делать дальше, чтобы закрыть gap и ты получил пользу.

### AB-BA Commutator Cross-Check (Mandatory when 2+ operations interact)

When delivery depends on two operations/paths (for example A=raw workflow, B=integration/output):

- Run **AB** and **BA** orders on the same input.
- Compute commutator deltas:
  - `Delta_offer = (A→B offer gap) - (B→A offer gap)`
  - `Delta_quality = (A→B quality gap) - (B→A quality gap)`
- If any Delta ≠ 0, mark as unstable and do not claim reproducible outcome.
- Save artifact path (json/report) in project todo and mention it in chat.

## Принцип «Шаг за шагом» (Goldratt): миграция документов и массовые правки

**Когда применять:** реструктуризация больших markdown-файлов (`changelog.md`, `<internal-folder>/ai.incidents.md`, реестры), выравнивание **одной** markdown-таблицы на календарный день (см. Standard 1.1 § канон `ai.incidents.md`), любой сценарий, где ошибка масштаба = потеря фактов или поломка парсеров.

**Канон `ai.incidents.md` (и аналогичные дневные журналы):** файл начинается с общего введения; дальше **одна секция уровня `##` на календарный день** (например `## 5 Apr 2026`), краткий текст дня при необходимости, затем **ровно одна** markdown-таблица на этот день. Все события этого дня (короткий trace и полный RCA) — **строки одной и той же таблицы** с **одинаковым заголовком** (одинаковое число колонок). Новые строки добавляются **сверху** таблицы дня (как сейчас по смыслу append-only внутри дня). Старые длинные блоки под `## 5 Apr 2026 — …` **переносятся** в строки таблицы этого же дня; подзаголовок дня может остаться как **человекочитаемый тезис** в колонке `title` или в первой строке секции, но не как вторая параллельная таблица с другой шириной.

**Суть (критическая цепочка):** узкое место — не скорость правок, а **доказуемое сохранение смысла**. Автоматизация расширяется только после того, как гипотеза «формат + скрипт не теряют поля» **не опровергнута на малых N** и **не опровергнута на растущих N** ([hypothesis-gap-falsification](.agents/skills/2-hypothesis-gap-falsification/SKILL.md)).

**Обязательная последовательность (не пропускать ступени):**

| Ступень | N (элементов) | Действие | Фальсификация |
|--------|----------------|----------|----------------|
| **S0** | 0 | Зафиксировать целевую схему колонок **одной** таблицы дня, правила экранирования ячеек (`\|` в тексте), канон дат (UTC в колонке времени). | Чеклист колонок в чате + ссылка на skill/стандарт (1.1, этот раздел). |
| **S1** | 1–2 | Вручную перенести 1–2 записи (или 1–2 старых RCA) в **целевой** формат таблицы **внутри одной дневной секции**. | [hypothesis-gap-falsification](.agents/skills/2-hypothesis-gap-falsification/SKILL.md): таблица «ожидание \| факт \| Δ» — все ли поля старой записи видны в новой строке? |
| **S2** | 0 | Написать или обновить **минимальный** скрипт (валидация числа колонок **по каждой таблице дня**, dry-run, отчёт расхождений). | Запуск на файле после S1: скрипт **PASS** на уже исправленных строках. |
| **S3** | 1–2 | Прогнать скрипт + дописать/мигрировать ещё 1–2 записи **полуавтоматом** (скрипт предлагает строку, человек правит). | Снова falsification: не появилось ли пустых обязательных колонок? |
| **S4** | — | Починить скрипт по инцидентам S3; при необходимости строка в таблице дня в `ai.incidents.md`. | RCA лёгкий: что сломалось в парсинге / экранировании. |
| **S5** | 5 | Мигрировать пачку из **5** записей; скрипт валидации на всех затронутых дневных таблицах. | Сравнение: число смысловых блоков до/после (commutator: не уменьшилось ли число зафиксированных фактов). |
| **S6** | 10 | Повторить на **10** записях. | То же + выборочное чтение 3 строк глазами не тем же агентом/сессией (если возможно). |
| **S7** | bulk | Массовая миграция остатка **только если** S5–S6 без потери полей и валидатор зелёный. | Полный прогон валидатора; при сбое — откат к последней хорошей ревизии git. |

**Запрещено:** прыгать сразу к «переписать весь файл», если не пройдены S1–S3. **Разрешено:** хранить длинный текст в одной ячейке с явным правилом переносов (например HTML `<br>` в GitHub-flavored таблицах) или вынести «хвост» в `reasoning-logs` с **обязательной** колонкой `detail_path` в **той же строке** дневной таблицы.

**Где ещё зафиксирована та же идея (критическая цепочка / Голдрат):** Standard [1.4 From-The-End](<standard-ref>), [1.7 Goldratt](<standard-ref>), [1.13](<standard-ref>) (строка про critical chain loss), [1.6 RCA](<standard-ref>) (TOC), [AI Incident 1.1](<standard-ref>) § канон `ai.incidents.md`, `.agents/skills/2-rca-incidents/SKILL.md`, `.agents/skills/1-task-completion-persistence/SKILL.md`, при необходимости ссылка из `.cursor/rules/core-auto.mdc` (Domain / planning).

## Reasoning Log Protocol (ОБЯЗАТЕЛЬНО)

При каждом исполнении этого скилла агент ОБЯЗАН:

1. **Вести reasoning log в чате** — таблица решений с evidence, gaps и blocking instructions:

```markdown
### Reasoning Log — [дата UTC]
| # | Decision | Evidence source | Gap found | Blocking instruction | Owner value |
|---|----------|-----------------|-----------|---------------------|-------------|
```

**§0 macro (hard fail):** колонка **Gap found** — не голый `G01`, а **`G01 — краткое имя гепа`**. Любой ID (`P0`, `G01`, `E01`, `pr-rick-*`) без ` — {человекочитаемое имя}` = hard fail. См. `AGENTS.md` секция «Plain language» и `CLAUDE.md` §0.

2. **Записать строку в `ai.incidents.md`** — в **дневной** секции `## DD MMM YYYY`, в **единственную** таблицу этого дня (см. Standard 1.1 § канон файла; колонки — по текущему заголовку таблицы дня, без второй таблицы с другой шириной):

```
| {UTC timestamp} | {skills} | {owner prompt ≤240} | {steering?} | {target artifact} | {reasoning bullets} | {blocking_instruction} | … |
```

3. **При задачах > 3 ходов** — сохранить лог в `<internal-folder>/reasoning-logs/`.

Hard fail: без reasoning log скилл считается неисполненным. См. протокол **agent-reasoning-log** в `AGENTS.md` (список навыков).
