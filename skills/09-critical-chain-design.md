# Critical Chain Design — проектирование критической цепочки Голдрата

## Hired for JTBD

Когда нужно спланировать задачу так, чтобы не было раздувания подзадач (inventory list), а была цепочка от результата к первому шагу → применить Theory of Constraints (теория ограничений Голдрата) для построения критической цепочки из 3-5 шагов.

## When to use

- Планирование любой задачи с >2 шагами
- «Спланируй от конца к началу»
- «Какое ограничение?», «Где узкое место?»
- Создание beads-карточки для нового проекта
- Когда подзадачи раздулись >5 штук — переосмыслить через constraint

## Формат цепи в outcome-designer субагенте: outcome → output → blockers

В outcome-designer карточке критическая цепь Голдратта записывается сверху вниз **от outcome к первому output**, с явным указанием узкого места (blockers). Это визуально показывает что каждый output-шаг — промежуточный результат, ведущий к единственному outcome:

```
Outcome     {измеримый outcome из замыкающего уравнения}
  ↑ output  {последний output перед outcome}
  ↑ output  {предпоследний output}
  ↑ output  {... 3-5 output-звеньев}
  ↑ output  {первый output из сегодняшнего состояния}
Blockers    {узкое место — какое звено bottleneck и почему}
Первый шаг  {конкретное действие сегодня}
Метрика     {binary или число с порогом}
```

Правило проверки: **последний output сверху обязан напрямую вести к outcome без недокументированного шага**. Если между верхним output и outcome есть «воздух» — добавить недостающее output-звено.

Canon: `.agents/agents/outcome-designer.md` формат карточки блок «Критическая цепь Голдратта».

## Core route — 5 шагов

### Шаг 1: Определить Outcome и Output

- **Outcome** = что изменится в мире после выполнения задачи (одна фраза)
- **Output** = конкретный артефакт, который мы доставляем

```
Outcome: {кто} может {что} без {чего лишнего}
Output: {конкретный файл / API / сообщение / документ}
```

### Шаг 2: Найти ограничение (constraint)

Ограничение = **один** ресурс или условие, без которого Output невозможен. Не список «всего что нужно», а **одно** узкое место.

Типы ограничений:
- **Данные:** нет входных данных для работы
- **Доступ:** нет API-ключа, нет прав, sandbox
- **Знание:** не понятна структура / формат / требование
- **Зависимость:** блокирован другой задачей
- **Время:** крайний срок диктует scope

### Шаг 3: Построить цепочку от конца

```
Output ← {что нужно для Output} ← {что нужно для предыдущего} ← ... ← Первый шаг
```

Каждое звено = **один наблюдаемый артефакт** (не «подумать» / «изучить» / «разобраться»).

### Шаг 4: Проверить длину (3-5 шагов)

| Шагов | Вердикт | Действие |
|-------|---------|----------|
| ≤2 | Слишком грубо | Декомпозировать Output на промежуточные артефакты |
| 3-5 | Хорошо | Продолжить |
| 6-8 | Перегруз | Вынести параллельные ветки; проверить, не раздул ли inventory |
| >8 | Inventory list | Вернуться к шагу 2, переопределить constraint |

### Шаг 5: Определить буфер и верёвку

- **Буфер** = запас времени/scope перед Output (обычно 20-30% от цепочки)
- **Верёвка** (rope) = механизм запуска работы по цепочке (не раньше, чем нужно)

```
Буфер: {что можно сократить / отложить, если constraint затянется}
Верёвка: {какой сигнал запускает следующий шаг}
```

## Anti-patterns (запрещено)

| Anti-pattern | Почему плохо | Что делать |
|---|---|---|
| Inventory list (список всего, что «надо бы») | Нет приоритета, нет constraint | Вернуться к шагу 2 |
| Fake constraint («всё важно одинаково») | Нет фокуса | Найти ONE thing без которого Output невозможен |
| Equal priority (все шаги P0) | Нет последовательности | Построить цепочку, а не матрицу |
| Unbounded discovery («сначала изучим всё») | Бесконечный scope | Определить Output, потом discovery только для gaps |

## Output contract

Финальный ответ содержит:

1. `Outcome` (одна фраза)
2. `Output` (конкретный артефакт)
3. `Constraint` (одно ограничение)
4. `Critical chain` (3-5 шагов от Output к первому шагу)
5. `Buffer` (что можно сократить)
6. `Rope` (сигнал запуска)

## Hard fail conditions

1. >8 шагов в цепочке без объяснения
2. Нет явного constraint (всё «одинаково важно»)
3. Шаги без наблюдаемых артефактов («изучить», «подумать»)
4. Inventory list вместо цепочки

## Owner value

Экономия усилий owner на ревью плана: вместо 15 подзадач видно 3-5 шагов с чётким constraint и buffer.

## Reasoning Log Protocol (ОБЯЗАТЕЛЬНО)

При каждом исполнении этого скилла агент ОБЯЗАН:

1. **Вести reasoning log в чате** — таблица решений с evidence, gaps и blocking instructions:

```markdown
### Reasoning Log — [дата UTC]
| # | Decision | Evidence source | Gap found | Blocking instruction | Owner value |
|---|----------|-----------------|-----------|---------------------|-------------|
```

2. **Записать строку в `ai.incidents.md`** — таблица `## Append-only trace`.

3. **При задачах > 3 ходов** — сохранить лог в `[todo · incidents]/reasoning-logs/`.

Hard fail: без reasoning log скилл считается неисполненным.
