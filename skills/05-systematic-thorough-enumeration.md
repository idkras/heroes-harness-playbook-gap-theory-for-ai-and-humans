# Systematic Thorough Enumeration Skill (Универсальный скилл настойчивости и системности)


## Hired for JTBD

Когда агенту нужно выполнить задачу, связанную с этим скилом → owner получает результат с reasoning log, evidence-based решениями и привязкой к value.



## Цель

Выполнять задачи, в которых нужно **много и упорно перебрать детали глазами** — по одному элементу, по порядку, без пропусков и без срезания углов. Универсален: подходит для сделок, строк таблицы, файлов, тикетов, комментариев, страниц документации и любого упорядоченного множества. Гарантирует: явное определение популяции, обработку по индексу/порядку, чекпоинты, возобновляемость и обязательную самопроверку на когнитивные искажения. Метрики устремления: все элементы 0..N обработаны по порядку, без пропусков, с reflection checkpoints каждые K элементов и финальным evidence блоком.

**Связано:** [research.mdc](.cursor/rules/research.mdc) (разделы «Когнитивные искажения AI агента», «Протокол анти-искажения», «Само-анализ когнитивных искажений»), [core-anti-bias.mdc](.cursor/rules/core-anti-bias.mdc) (Attribution, Counter-Evidence, Missing Data, Alternatives, Ignorance).

---

## Когда применять

- Задача явно или неявно требует **перебрать много однотипных элементов** (сделки, строки CSV, файлы в папке, комментарии, тикеты, пункты списка).
- Есть риск **cherry-picking**: взять «несколько штук» или «выборку» и сделать вывод по ним — тогда вывод будет смещён (survivorship bias, confirmation bias).
- Пользователь говорит: «все изучить», «не выборочно», «систематически», «там в разы больше», «много и упорно перебрать детали глазами», «не чекри пикингом».
- Нужна **возобновляемость**: обработали 0–500, потом 500–1000 и т.д. без потери места.

**Примеры задач (не только данные):** разбор всех сделок из выгрузки; проверка всех файлов в директории; проход по всем комментариям к тикету; вычитывание всех пунктов договора; инвентаризация всех полей в форме.

## Когнитивные искажения, которые этот скилл предотвращает

Перед началом и на чекпоинтах обязательно сверяться с правилами (research.mdc, core-anti-bias.mdc):

### 1. Survivorship bias (ошибка выжившего)

- **Суть:** видишь только «успешные» или «удобные» элементы, не видишь остальные → неверная картина.
- **Как возникает у агента:** неявный limit (взял первые 10/50), анализ только «заполненных» полей, пропуск «пустых» или «скучных» строк, фокус на «типичных» случаях.
- **Обязательное правило:** обрабатывать элементы **строго по порядку** в заданном диапазоне `[start_index, end_index)`; не ставить лимит по умолчанию; пустые/ошибочные элементы тоже учитывать (фиксировать как «пусто» или «ошибка»), не пропускать молча.

### 2. Confirmation bias (предвзятость подтверждения)

- **Суть:** ищешь только то, что подтверждает гипотезу, игнорируешь противоречащее.
- **Как возникает:** фильтрация по ожидаемым полям, выбор «интересных» записей, прекращение перебора после первых «подходящих».
- **Обязательное правило:** для каждого элемента выполнять **одну и ту же процедуру** (прочитать все заданные поля/части, применить один и тот же критерий); не останавливаться по достижении «достаточного» числа примеров; в отчёте явно писать, что противоречит основной гипотезе.

### 3. Ложная mental model

- **Суть:** строится неверная модель на основе неполного перебора.
- **Как возникает:** вывод по первым 20 строкам, предположения о структуре без прохода по всем элементам.
- **Обязательное правило:** явно определить **полную популяцию** (источник, общее количество, порядок); обрабатывать только заданный диапазон, но не выдавать вывод по «всей» популяции, пока не обработан весь запланированный объём или не указано ограничение («обработано 0–500 из N, вывод только по ним»).

### 4. Cherry-picking (выборочный отбор)

- **Суть:** сознательно или неосознанно выбираешь «удобные» элементы вместо систематического прохода.
- **Обязательное правило:** **никакой выборки по содержанию** (кроме явно заданной пользователем, с описанным методом). Единственный допустимый способ — по **индексу/порядку**: обрабатываю 0, 1, 2, …, N-1 в заданном диапазоне.

## Универсальный алгоритм (без привязки к домену)

### Шаг 0. Подготовка (обязательно перед первым элементом)

1. **Определить популяцию:**
   - Источник (файл, папка, API, список).
   - Общее количество элементов N (или способ его получить: число строк, число файлов и т.д.).
   - Порядок (индекс 0, 1, 2, … или явный порядок: по дате, по имени).
2. **Определить «один элемент»:** что считается элементом (одна строка, одна сделка, один файл, один комментарий).
3. **Определить процедуру на один элемент:** что именно делать с каждым (прочитать поля A, B, C; извлечь X, Y, Z; записать в выход одну запись).
4. **Задать батч и чекпоинты:**
   - Диапазон этого запуска: `[start_index, end_index)` (например 0–500). Если продолжение — пользователь или контекст задаёт start_index.
   - Чекпоинт каждые K элементов (например 50 или 100): вывести прогресс, проверить, что не пропущены индексы, выполнить самопроверку на искажения.
5. **Выход:** куда писать результат (файл, таблица) и в каком формате (одна запись на элемент).

**Запрещено:** начинать перебор без явного N или без явного диапазона; вводить «выборку» или «limit 10» без явного решения пользователя.

### Шаг 1. Обработка по порядку

Для каждого индекса `i` в `[start_index, end_index)`:

1. Получить элемент с индексом `i` (строка i, файл i, сделка i и т.д.).
2. Применить к нему **одну и ту же** процедуру (все заданные поля прочитать, все шаги выполнить). Пустые поля фиксировать как пустые, не пропускать.
3. Записать результат по элементу в выход (одна строка/один JSON/одна запись).
4. Не переходить к следующему элементу, пока не завершена обработка текущего.

**Запрещено:** пропускать индекс; обрабатывать только «интересные» элементы; менять процедуру в середине батча; ставить внутренний limit «взять 20 и хватит».

### Шаг 2. Reflection checkpoint (каждые K элементов)

- Вывести в чат: обработано элементы с `start` по `current`, всего записей в выходе, краткая сводка (сколько пустых, сколько с ошибкой, примеры).
- Проверить: для всех ли индексов в отрезке есть запись в выходе? Нет ли дублей по индексу?
- **Самопроверка на искажения:**
  - Survivorship: не анализирую ли я только «успешные» записи? Учитываю ли пустые и ошибочные?
  - Confirmation: не подгоняю ли выбор под гипотезу? Есть ли в выводе противоречащие примеры?
  - Mental model: не делаю ли вывод по «всей» популяции, пока обработан только батч?

### Шаг 3. Итог по батчу

- Сколько элементов обработано (число).
- Путь к выходному файлу/артефакту.
- Если `end_index < N`: явно написать: «Для продолжения задать start_index = end_index» (или следующий диапазон).
- **Ограничения:** что не обработано, что не проверено, по чему вывод делать рано.

### Шаг 4. Валидация (рекомендуется)

- Выборочно открыть 2–5 записей из выхода и сверить с исходным элементом (тот ли индекс, те ли поля).
- Зафиксировать в чате или в `{projectname}.todo.md`: батч выполнен, метрики, ссылка на артефакт.

---

## Обязательная самопроверка перед формулировкой «готово»

Перед тем как написать пользователю, что перебор выполнен:

- [ ] **Survivorship:** Обрабатывал ли я все элементы в диапазоне подряд? Не ставил ли скрытый limit? Учитываю ли пустые/ошибочные элементы?
- [ ] **Confirmation:** Применял ли одну и ту же процедуру ко всем? Не отбирал ли «подходящие» под гипотезу? Есть ли в отчёте противоречащие факты?
- [ ] **Mental model:** Явно ли указано, по скольким элементам вывод (батч 0–500 или полный N)? Не обобщаю ли на всю популяцию по неполным данным?
- [ ] **Cherry-picking:** Не выбирал ли элементы по содержанию? Только ли по индексу/порядку?

Если хотя бы один пункт не выполнен — поправить процесс или явно написать в отчёт ограничения и что не сделано.

## Входные параметры (универсальные)

| Параметр | Описание |
|----------|----------|
| **source** | Откуда брать элементы (путь к файлу, папке, идентификатор списка) |
| **item_definition** | Что считается одним элементом (строка CSV, файл, сделка и т.д.) |
| **procedure_per_item** | Что делать с каждым элементом (список полей, шагов или ссылка на чеклист) |
| **start_index** | С какого индекса начать (0 для первого запуска) |
| **end_index** или **batch_size** | До какого индекса обработать или сколько элементов в батче |
| **checkpoint_every** | Каждые N элементов — reflection checkpoint |
| **output_path** | Куда писать результат (файл, таблица) |

Домен-специфичные параметры (например «путь к CSV компаний для джойна») задаются в применении скилла к конкретной задаче (см. пример ниже).

## Пример применения: сделки AmoCRM

Для задачи «обработать все сделки из CSV, подтянуть компании и ИНН, извлечь этапы CJM и геп-фразы»:

- **Популяция:** строки CSV сделок, N = число строк, порядок по индексу строки.
- **Один элемент:** одна строка (сделка).
- **Процедура на сделку:** прочитать все текстовые поля (Примечание 1–5, Звонок, Доведение до второй покупки, Комментарии, комментарии к телефону/почте/имени); склеить в один текст; по полю «Компания» найти в CSV компаний ИНН; распарсить этапы CJM и геп-фразы; записать одну запись в JSONL/CSV.
- **Батч:** например 500 сделок за запуск; чекпоинт каждые 100.
- Полный чеклист полей и парсинга для AmoCRM — в задаче клиента (autovin todo) или в скилле-приложении [amocrm-deals-systematic-extraction](.agents/skills/8-amocrm-deals-systematic-extraction/SKILL.md).

Универсальный скилл задаёт **как** перебирать (по порядку, без пропусков, с чекпоинтами и самопроверкой). Домен-специфичный чеклист задаёт **что** читать и что писать для сделок.

## Output Checklist

- [ ] Популяция определена (источник, N, порядок).
- [ ] Обработаны **все** элементы в диапазоне `[start_index, end_index)` по порядку, без пропусков.
- [ ] Для каждого элемента применена одна и та же процедура; пустые/ошибочные зафиксированы, не проигнорированы.
- [ ] Reflection checkpoint выполнен каждые K элементов; в чат выведен прогресс и самопроверка на искажения.
- [ ] Итог по батчу: число обработано, путь к выходу, при необходимости — как продолжить (start_index = end_index).
- [ ] Перед «готово» выполнена обязательная самопроверка (survivorship, confirmation, mental model, cherry-picking).

## Reasoning Log Protocol (ОБЯЗАТЕЛЬНО)

При каждом исполнении этого скилла агент ОБЯЗАН:

1. **Вести reasoning log в чате** — таблица решений с evidence, gaps и blocking instructions:

```markdown
### Reasoning Log — [дата UTC]
| # | Decision | Evidence source | Gap found | Blocking instruction | Owner value |
|---|----------|-----------------|-----------|---------------------|-------------|
```

**§0 macro (hard fail):** колонка **Gap found** — не голый `G01`, а **`G01 — краткое имя гепа`**. Любой ID (`P0`, `G01`, `E01`, `pr-rick-*`) без ` — {человекочитаемое имя}` = hard fail. См. `AGENTS.md` секция «Plain language» и `CLAUDE.md` §0.

2. **Записать строку в `ai.incidents.md`** — таблица `## Append-only trace`:

```
| {UTC date} | {skill_name} | {owner prompt ≤240} | {steering: yes/no} | {target artifact} | {reasoning bullets} | {blocking_instruction} |
```

3. **При задачах > 3 ходов** — сохранить лог в `[todo · incidents]/reasoning-logs/`.

Hard fail: без reasoning log скилл считается неисполненным. См. протокол **agent-reasoning-log** в `AGENTS.md` (список навыков).
