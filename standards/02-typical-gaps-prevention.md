## 🎯 Цель документа

Свести в один справочный стандарт **типовые гэпы и проблемы AI-агента**, из-за которых пользователю приходилось направлять и исправлять работу. Источники: правила `.cursor/rules/core-auto.mdc` (собранные по инцидентам протоколы) и лог инцидентов `<internal-folder>/ai.incidents.md`. Цель — дать агенту короткий чеклист «чего избегать» и «что проверять», чтобы снижать количество инцидентов и повторных направлений (Definition of Done - DOD).

## 📋 Типовые гэпы и проблемы (таблица)

| **Existing-ticket rewrite / enforcement drift** | Агент materialize-ит или продолжает existing bead после owner feedback `плохой тикет`, но сам bead не переписан по полному канону; параллельно rewrite route в skill указывает на stale/missing helper, а checker остаётся зелёным на более слабом контракте | Для bad existing ticket agent должен: показать standard violations -> запустить same-ticket rewrite через живой `ticket-review-update` -> сделать writeback -> проверить readback; checker должен валидировать route на реальный rewrite skill, `5 почему`, post-change loop и redo-verdict markers | ai.incidents 30 Mar 2026 |

| **Runtime contour mismatch / optional dependency assumption** | Агент или MCP-tool считает, что declared dependency уже есть в runtime contour, и начинает разбирать symptom как data/API bug; parallel preview path при этом может маскировать реальный execution blocker generic ошибкой | Для runtime-sensitive Rick workflows сначала отделять `repo requirements`, `local supported runtime`, `actual MCP process contour`; optional deps (`aiofiles` и аналоги) должны иметь fallback path, а preview/diagnostic tools обязаны показывать canonical живой endpoint вместо generic error | ai.incidents 27 Mar 2026 |

## 🔗 Связь с протоколами и скиллами

- **core-auto.mdc** — основные протоколы (NO NEW DOCUMENTS, DUPLICATE CHECK, PARAMETER VERIFICATION, OUTPUT/OUTCOME, ITERATIVE TESTING, CLIENT-PERSPECTIVE, REFLECTION CHECKPOINTS и др.).
- **core-check.mdc** — верификация результата, калибровка уверенности, gap analysis, Challenge Protocol.

- **rca-incidents skill** — при работе по инциденту: фиксация в ai.incidents.md → RCA → дизайн-инъекция → цикл изменений → **обновление этого стандарта (таблица сводки и при необходимости таблица типовых гэпов)** → новая доставка.
- **standards-create-update-review skill** — при создании/обновлении стандартов: обязательное обновление Registry 0.1 и фиксация в реестре.

## ✅ Рекомендации по снижению инцидентов (операционный чеклист)

Как агенту работать, чтобы было меньше инцидентов и повторных направлений:

1. **В начале сессии или перед рискованным действием** — перечитать этот стандарт (хотя бы таблицу типовых гэпов) и таблицу инцидентов.
2. **Перед созданием любого файла** — NO NEW DOCUMENTS: проверить document-creation-guard; если destination не определён или документ запрещён → писать в чат / `.beads`, выбрать local folder и не использовать root `docs/` / не воссоздавать этот sink.
3. **Перед созданием скрипта/MCP/нового кода** — DUPLICATE CHECK: семантический поиск → отчёт в чат → не создавать при confidence <0.7.
4. **Перед отправкой в чат команде (дейли, доставка)** — черновик → review-artifact-for-client-readiness → правки → отправка.
5. **Во всех substantial промежуточных апдейтах** — явно писать `Блокер`, `Решаю`, `Чтобы что`. Если работа идёт с nested repo, submodule, fork, external MCP или unpublished SHA — отдельной строкой писать ownership/publishability blocker.
6. **При выдаче ссылок на виджеты/группы <internal-component>** — только `generate_widget_url` / `generate_group_url`; при ошибке — инструкция по открытию в UI, не строить URL вручную.
7. **Перед sync-github / коммитом путей** — проверка имён на запрещённые в Windows символы (`\ / : * ? " < > |`).
8. **Перед инициализацией API/клиента** — Parameter Verification: проверить наличие всех required параметров.
9. **После изменения кода** — один шаг → найти тесты → запустить → зафиксировать результат; не помечать задачу ✅ без проверки.
10. **После любого вывода** — калибровка уверенности: `Confidence: X% → Y% — причина`.
11. **При доставке артефакта** — явно писать в чат ссылку (URL, путь, document ID); для Outline — публичная ссылка для клиента.
12. **При сбое MCP/внешнего API** (например n8n, <internal-component>): явно написать в чат, что артефакт **не прочитан / не обновлён** (API недоступен). Если артефакт может быть в репо (экспорт workflow, JSON, сохранённые копии) — попытаться прочитать и сообщить текущее состояние; не оставлять впечатление, что «ноду проверил», если не проверял.
13. **Перед вызовом Telegram MCP** (search_chats_by_keyword, get_all_chats_list, get_direct_chat_by_contact): применить скилл telegram-mcp-launch; всегда передавать **limit** (например 20–50) для search/get_all; **не вызывать два тяжёлых инструмента параллельно** — выполнять по одному вызову за раз.
14. **При вызове скилла по имени** (например «skill next», «скилл next»): выполнить **все шаги** этого скилла в этой же сессии (не использовать скилл только как справочник); для скилла next — в конце явно написать доставку результата и **«Польза тебе: да / нет»** (ai.incidents 15 Feb 2026).
15. **При блокере «применить миграцию» или «запустить скрипт X»** (разблокирующий шаг): **сначала запустить** этот скрипт в сессии (env или Keychain); только если скрипт падает из‑за отсутствия кредов — явно написать в чат и попросить добавить (Keychain / env). Не писать «нужно сделать вручную» без попытки запуска (ai.incidents 15 Feb 2026, Telegram migration).
16. **При запросе «полная синхронизация»** тяжёлого скрипта (напр. sync_telegram_chats_to_supabase, выполнение 1–5+ мин): **не запускать** один run_terminal_cmd с большим лимитом (--max-dialogs 1000 и т.п.) в Cursor — риск Aborted. Запускать **по этапам** (напр. --limit 200, затем 400) или **выписать команду и инструкцию** для запуска пользователю во внешнем терминале (ai.incidents 15 Feb 2026, anti-freeze).
16. **Перед анализом виджетов Rick для JTBD-вопроса** (напр. «какие города покупают», «киты по транзакциям»): открыть `<standard-ref>) и типовые группировки; затем `get_widget_groups` → искать группу по id; по `params.groups` виджета выбрать нужный. Различать: «папка в Rick» = widget group; «scenario-folder с файлами» = папка в Drive/репо (ai.incidents 16 Feb 2026).
17. **Если sync-github повторно упёрся в тот же blocker** (`changelog.md is not staged`, mutating `pre-push`, nested repo `403`): **не повторять** `git add -A -> git commit -> git push`. Сначала применить `sync-github-checklist` `Critical loop-breakers`: RCA, changelog update, hook inspect, publish nested repo SHA на remote/fork, затем один финальный `stage all -> commit -> push`.
17.1. **Если пользователь отдельно просит `stage all` / `git add -A`**: сначала классифицировать это как `local snapshot` или `publishable selective commit`. В грязном монорепо `stage all` по умолчанию означает только локальный snapshot; publish scope собирается selective `git add <paths>`. Старый staged snapshot нельзя автоматически считать следующим commit payload.
18. **Если nested repo commit нужен в parent repo:** перед обновлением gitlink проверить, что commit опубликован на remote. При `403` на upstream — создать/использовать fork и пушить туда; parent repo не должен ссылаться на локальный-only SHA.
19. **Если `git push` уже успешен и remote SHA подтверждён**, а после этого появились свежие `*_latest` / snapshot-файлы: не считать это провалом sync. В отчёте отделить `published commit` от `new residual dirty tree`; новый sync запускать только по explicit ask пользователя.
20. **При отладке сбоев get_widget_data или <internal-component> API** (HTTP 500, неверные даты, «лажовые параметры»): проверить **логи MCP сервера <internal-component>** (процесс/файлы логов <internal-component>_mcp) на **фактические переданные параметры** (start_date, end_date, company_alias, app_id, widget_id). В `<internal-component>_mcp_server.py` каждый вызов get_widget_data логируется с этими параметрами (logger.info) — по логам видно, что реально ушло в вызов. Не делать вывод «API вернул ошибку» без проверки логов на переданные значения (ai.incidents 17 Feb 2026).
21. **Перед вызовом get_widget_data:** убедиться, что в **параметрах вызываемого инструмента** (не в тексте сообщения в чат) start_date и end_date заданы **строками формата YYYY-MM-DD** (например 2026-01-01). Не передавать только год (2026) или опечатку (2826). Самопроверка: «что я реально подставляю в вызов? Совпадает ли с тем, что я написал пользователю?» (ai.incidents 17 Feb 2026, рекуррент — агент заявил «корректные даты», отправил 2026.)
22. **При задаче «PraxisGPT анализ» / «по стандарту v1.8»:** открыть Standard v1.8; проверить наличие **URL лендинга** или **client texts** (JTBD graph, FAQ и т.п.). Если есть URL — вызвать MCP `heroes_gpt_workflow(url)`. Если нет URL, но есть client texts — положить в `{domain}_client_texts/` и вызвать `heroes_analyze_landing` (workflow_integration). **Запрещено:** создавать документ вручную (read+write .md), имитирующий структуру PraxisGPT без запуска workflow (ai.incidents 19 Feb 2026, <client>).
23. **После создания значимого артефакта** (документ в `<standard-ref>): применить project-todo-registry-check; если задача трассируема — добавить строку в реестр todo.md или создать `{projectname}.todo.md`. Не оставлять артефакт «потерянным» в проектах (ai.incidents 19 Feb 2026, Constitution).

30. **Если ответ должен разблокировать ручное действие пользователя здесь и сейчас**: начинать финальный ответ с `Action Contract` прямо в чате: `Уже сделано`, `Сделай сейчас (1-3 шага)`, `Не делай`, `Остаётся заблокировано и почему`.
