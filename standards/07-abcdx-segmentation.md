## 🧠 ABCDX — сегментация по ценности продукта (автор И. Красинский)

**Источник:** методология разработана Ильёй Красинским; изложение и кейсы — [Иван Замесин, ABCDX-сегментация](https://zamesin.ru/books/product-howto/abcdx-segmentation/). Для advising и анализа разговоров используем **интерпретацию автора** (ниже).

### Определения сегментов (базовый подход автора)

- **A сегмент** — текущий продукт и сервис уже подходят клиенту; даже не идеальные, даже с проблемами, но под этих клиентов подходят.
- **B сегмент** — мы немного для них не подходим: не тот процесс продаж, или частично не хватает возможностей.
- **C сегмент** — с ними сложно: своя объяснительная модель, тяжело найти общий язык.
- **D сегмент** — им объективно дорого, ценности не видят (например, физически нет этой суммы).
- **X сегмент** — потенциально наш A, но мы до них не доросли: не хватает бренда, имени, доверия, процессов продаж или возможностей.


Для анализа звонков и разговоров по JTBD-сценариям и для ABCDX-сегментации разговоров **используем определения автора** (A/B/C/D/X выше); при необходимости можно сопоставлять с формулировками Замесина.

### Квалифицирующий вопрос при сегментации (авторство: Xander, Александр Кузницов)

Перед отнесением ситуации к сегменту A/B/C/D/X задать вопрос:

**«Эта ситуация (клиент не покупает / сделка не двигается) — это про нас, про наш процесс и нашу работу или про клиента?»**

- **Про нас / наш процесс:** сбой в процессе продаж, коммуникации, сроках, качестве подстройки под клиента → чаще ведёт к сегментам **B** (мы немного не подходим) или **X** (мы не доросли). Действия: улучшать процесс, обучение, чек-листы, связность этапов.
- **Про клиента:** объективные ограничения клиента (бюджет, сроки, выбор другого поставщика, контекст) или несовпадение ценности → **D** или **C**. Действия: не перегружать процесс, сегментировать оффер, не тратить лишний Effort₂.

Использование: при разборе сделок и примечаний сначала классифицировать по этому вопросу, затем присваивать A/B/C/D/X — чтобы не списывать на «плохой клиент» то, что является гепом процесса (см. [Gap Theory Standard](abstract://standard:gap_theory_standard)).

## 🔄 Методы и алгоритмы: прохождение по сделке, гепы и застревания → ABCDX

**Идея:** Сам факт того, **как клиенты проходят по сделке** — где цепляются за процесс, в каких местах возникают гепы и дырки, где застревают — разбивает клиентов по сегментам A/B/C/D/X. Сегментация не только по «кто клиент», но и по **траектории в процессе**: движение к оплате vs застревание на этапе, триггеры (сделка движется) vs анти-триггеры (блокер, риск потери).

**Связь с теорией гепов:** По [Gap Theory Standard](abstract://standard:gap_theory_standard) — процессный, временной, интерпретационный, JTBD- и контекстный гепы снижают throughput; застревание на этапе (ожидали артефакт — не получили, ожидали ответ — тишина) интерпретируется через квалифицирующий вопрос Xander (Александр Кузницов): «про нас» → B/X, «про клиента» → D/C.

### Двухфазный метод выявления сегментов и блокеров

| Фаза | Кто | Действие | Результат |
|------|-----|----------|-----------|
| **1. Сессия с селзом (эксперт)** | Адвайзер + селз (напр. менеджер по продажам) | Разобрать **~20 сделок** по тегам и этапам воронки; обсудить, где «цепляются», где теряем, почему сделка не двигается. | **Корневые блокеры продаж** по симптомам из головы селза: теги, этапы, типичные формулировки в примечаниях; гипотезы «симптом → что это значит». |
| **2. Чтение деталей выгрузки** | Адвайзер / аналитик | Читать поля выгрузки (этап, теги, примечания 1–5, хронология «Звонок», дата след. контакта); искать фразы и паттерны по списку блокеров и по этапам CJM. | **Телс-сигналы и квалификация:** доп. триггеры/анти-триггеры, маппинг этап/тег/фраза → сегмент ABCDX и тип гепa; таблица «симптом → гипотеза → как работать» для команды продаж. |

После фазы 2 правила можно формализовать: по каким полям и значениям относить сделку к A, B, C, D, X; чеклист телс-сигналов для разметки следующих выгрузок.

### Поля для анализа (CRM / выгрузка сделок)

Использовать для сегментации и квалификации. **Важно:** состав полей зависит от CRM и клиента; ниже — универсальные типы полей и примеры по системам (AmoCRM, Битрикс24, и т.д.).

| Поле | Назначение | Примеры использования для сегментации и квалификации |
|------|------------|------------------------------------------------------|
| **Этап сделки** | Стадия в воронке → близость к A (оплачено, договор) vs отказ/неактуально; подэтапы — анти-триггеры или запросы (B). | «Договор подписан» → A при движении; «Договор подписан. у нас дороже» → D; «Ждем реквизиты» → B; «не берёт» → D. Квалификация: по подэтапу и датам — сколько дней на этапе. |
| **Теги** | Предварительная разметка и блокеры. | VIP, ИП, ООО, «Пока не интересно», «у нас дороже», «Фин. проблемы», «Потеряли ЛПРа», «Офер пока не интересен». Сегментация: тег → маппинг в A/B/C/D/X; теги «не интересно»/«дороже» → D. |
| **Примечание 1–5** | Краткие ремарки селза по сделке. | Поиск фраз-триггеров («Отправила ДП», «настройка логина») и анти-триггеров («не берет», «бросил трубку»). Квалификация: последнее примечание с датой → свежесть контакта. |
| **Хронология общения (домен-специфичное поле)** | Единый текст или лог контактов по сделке — **название и структура зависят от CRM/клиента.** | **<client> (AmoCRM):** поле «Звонок» — хронология звонков и действий (даты DD.MM, фразы «не берет», «логин», «на почту»). **Другие клиенты:** могут быть «Активности», «История изменений», «Комментарии к задаче», «Лента событий», «Activity log» — везде ищем те же триггеры/анти-триггеры и даты. Использование: парсинг этапов CJM, подсчёт effort (число контактов), выявление блокер-фраз. |
| **Дата след. контакта** | Планируемый следующий шаг. | В будущем → триггер (запланировано общение); пустая или в прошлом при «Договор подписан» / «Ждем реквизиты» → анти-триггер, риск потери. Квалификация: разница с сегодня → приоритет обзвона. |
| **Дата создания / редактирования / закрытия** | Длительность цикла и свежесть сделки. | Дата создания и закрытия → Lead Time сделки (короткий цикл → ближе к A). Дата последнего редактирования при отсутствии движения → косвенный признак застревания (требуется история изменений для точного анализа). |
| **Компания, контекст** | Тип и масштаб клиента для интерпретации сегмента. | ИП/ООО/АО по названию или по обогащению (ИНН, оборот); «Свой автопарк»/«Чужой», «Количество авто» — контекст для B/D/X. **<client>:** тип CTO (иномарки vs коммерческий транспорт) — см. раздел «<client>: сегментация по amoCRM — CTO и типы транспорта». Сегментация: VIP/крупный оборот → приоритет A или X. |
| **Ответственный / Менеджер сделки** | Кто ведёт сделку. | Для анализа блокеров по менеджерам; для сессии с селзом (разбор «своих» сделок). Не для прямого маппинга A/B/C/D/X. |

**Примеры полей «хронология общения» в других CRM (аналог «Звонок» у <client>):**

- **AmoCRM:** «Звонок», «Доведение до второй покупки», «Комментарии» (тип поля может отличаться по воронке).
- **Битрикс24 CRM:** «История лида/сделки», «Активности», «Входящие/исходящие звонки», комментарии к этапу.
- **HubSpot:** Activity timeline (calls, emails, notes), engagement.
- **Excel/CSV выгрузка:** одна колонка с объединённым текстом «все примечания + лог» или несколько колонок с датой и текстом события.

Во всех случаях цель одна: извлечь **даты контактов** и **формулировки** (триггеры/анти-триггеры) для маппинга в сегмент и тип гепa. При отсутствии единого поля хронологии — склеить примечания 1–5, комментарии к этапу и т.д. в один текст и парсить по тем же правилам.

### Триггеры (сделка движется, не в блокере)

- Конкретные действия в примечаниях: «Отправила ДП», «настроили», «настройка логина», «настроить ЭДО», «Написала в телеграмм», «согласовать договор-отсрочки», «Рассматривают подключение API».
- Этапы: «Ждем реквизиты», «Выставили счет, ждём оплаты», «1 сделка оплачена», «2+ сделки оплачены» → движение к оплате.
- **Дата след. контакта** в будущем → запланировано общение.
- **Ответственный / Менеджер сделки** (телс по людям): кто вёл сделку до 1-й и 2+ оплаты; распределение по менеджерам по этапам (1 сделка оплачена, 2+ оплачены, Договор подписан) — выявляет, у кого выше доля доведённых до оплаты и какие приёмы/теги/заполненность полей отличают их сделки (см. раздел «Сегмент A vs застрявшие» ниже).

**Гипотеза для проверки:** формулировки-триггеры в примечаниях («Отправила ДП», «настройка логина», «настроить ЭДО» и т.д.) могут совпадать со сделками, которые затем **застревают на этапе на несколько дней или недель** (ожидание ответа клиента, следующий шаг не сделан). Верификация требует **истории изменений сделки** (когда этап менялся, когда добавлялось примечание): по одной выгрузке-снимку можно лишь считать «дней с последнего редактирования» или «дата след. контакта просрочена»; для связи «триггер → N дней на этапе» нужна таблица событий по сделке (см. проект Deal History → CFD). **Проверка по снимку (<client>, 2026-02-13):** у сделок «Договор подписан» с триггером в тексте среднее «дней с последнего редактирования» *меньше*, чем у сделок без триггера (3,4 vs 5,4 дня), т.е. по снимку гипотеза «триггер → застревание» не подтверждается (скорее триггер сопутствует недавней активности). Окончательная проверка — по таблице событий (дата примечания с триггером → дни до следующего события).

### Анти-триггеры (блокер, тормозят сделку / теряем лида или клиента)

- **Теги:** «Пока не интересно», «у нас дороже», «Фин. проблемы, не хотим работать», «Потеряли ЛПРа-ЛВРа», «Офер пока не интересен».
- **Этапы:** «не берёт», «Пока неактуально», «Договор подписан. Офер пока не интересен», «у нас дороже», «неправильный номер», «Компания ликвидируется».
- **Текст в примечаниях:** «не берет», «бросил трубку», «Не отвечают» / «не отвечает», «перестали покупать», «Компания ликвидируется», «у нас плохой сервис», «предпочитает обслуживаться в сто».
- **Дата след. контакта** в прошлом или пустая при этапе «Договор подписан» / «Ждем реквизиты» → риск потери: общение не возобновлено.
- **Ответственный / Менеджер сделки** (телс по людям): при анализе блокеров и анти-триггеров — по какому менеджеру больше застрявших сделок, у кого чаще пустая/просроченная «Дата след. контакта», какие формулировки в примечаниях повторяются; сравнение с менеджерами, у которых выше доля сделок «1 сделка оплачена» и «2+ сделки оплачены» (см. раздел «Сегмент A vs застрявшие»).

### Сегмент A (сделки 1 и 2+ оплачены) vs застрявшие: какие особенности смотреть

**Цель:** Понять, чем клиенты и сделки, которые **при текущем процессе дошли до 1-й и 2+ оплаты** (конец воронки, сегмент A), отличаются от тех, кто застрял на «Договор подписан» / «Ждем реквизиты» и заблокировался. Это даёт телс-сигналы для усиления процесса и чеклистов.

**Что сравнивать (поля и метрики):**

| Что смотреть | Зачем |
|--------------|--------|
| **Ответственный / Менеджер сделки** | Распределение по этапам: у кого выше доля сделок «1 сделка оплачена» и «2+ сделки оплачены» vs «Договор подписан» без движения; дифференциация практик между менеджерами. |
| **Теги** | VIP, ИП, ООО, импорт_… — частота у A vs у застрявших; теги-анти-триггеры («Не берет», «Пока не интересно») реже у доведённых до оплаты. |
| **Дата след. контакта** | У застрявших чаще пустая или в прошлом при «Договор подписан»; у доведённых до 2+ оплаты может быть реже заполнена (сделка уже в поддержке/повторах). |
| **Триггеры в тексте (Примечание, Звонок)** | В сделках A чаще встречаются фразы движения («оплатил», «заказ получили», «ДО», «выполнено»); в застрявших — анти-триггеры («не берет», «не отвечает», «ждем реквизиты» без следующего шага). |
| **Объём контакта (длина «Звонок», число касаний)** | Effort: у части застрявших — много касаний без перехода этапа (процессный геп); у части A — достаточная последовательность шагов. |

**Пример по выгрузке <client> B2B (2026-02-13):** «1 сделка оплачена» 79, «2+ сделки оплачены» 612, «Договор подписан» 555. По полю «Ответственный»/«Менеджер сделки»: у части сделок в конце воронки один и тот же ответственный (в т.ч. системный импорт); у «Договор подписан» — те же менеджеры с большей долей застрявших без даты след. контакта. Для верификации: выборочно разобрать 20–50 сделок A (1 и 2+ оплачены) по менеджеру — какие действия и теги в карточке; сравнить с 20–50 застрявшими «Договор подписан» с пустой/просроченной датой след. контакта. Результат: таблица «особенности A vs застрявшие» и при необходимости дополнение чеклиста блокеров и триггеров по менеджерам.

### Маппинг этап / тег / фраза → сегмент ABCDX (пример)

| Сигнал | Сегмент | Тип гепa (при застревании) |
|--------|---------|----------------------------|
| «2+ сделки оплачены», «Договор подписан» (без анти-триггеров), триггеры в примечаниях | **A** | — |
| «Ждем реквизиты», «1 сделка оплачена», «Нужна доставка день в день» (запрос по условиям) | **B** | Процессный: не подходим по условию/процессу |

| «неправильный номер», «Офер пока не интересен» (сложно общий язык / не тот продукт) | **C** | JTBD, интерпретационный |
| «не берёт», «Пока неактуально», «у нас дороже», «Фин. проблемы» | **D** | Временной, процессный; ценность не совпадает |
| «Первичный контакт», «Хочет работать по API» (потенциал при движении) | **X** | При отсутствии движения — анти-триггер, B/X |

### <client>: сегментация по amoCRM — CTO и типы транспорта

Для <client> (B2B запчасти, amoCRM B2B) дополнительная сегментация по **типу CTO и обслуживаемому транспорту** — определяет «наш» vs «не наш» сегмент и приоритеты в работе с лидами.

| Сегмент | Тип CTO | Транспорт | Конкуренты | Услуга (срочность) | Fit <client> | Обоснование |
|---------|---------|-----------|------------|--------------------|-------------|-------------|
| **D (не наш)** | CTO с иномарками | Импорт, иномарки | Авторусь, Роско, Артек | Доставка несколько раз в день; день в день; дарксторы по Москве; сокращение времени 2–3 часа | ❌ Не подходим | Огромные инвестиции в логистику; конкуренты уже закрывают потребность; мы никогда не дадим такой уровень срочности — не наш сегмент. |
| **A/B (наш)** | CTO с коммерческим транспортом | Отечественный, около отечественный, средний, крупный, иностранный | КэтЛоджистикс | Срочная, но след. день или день в день | ✅ Подходим | Рынок менее плотный и конкурентный; клиенты вынуждены ждать, альтернатив мало — мы можем играть. |

**Подсегмент «дороже, но быстрее/удобнее через <client>»:**

Иногда клиентам **быстрее и удобнее** прийти через <client>, даже если дороже. География: в основном **Москва, Московская область, Север**. Для этих регионов — приоритет офера «доставка день в день» / «след. день»; тег или пометка в amoCRM: `<client>_удобнее_дороже` для аналитики.

**A1 vs A2 — критерии разделения внутри сегмента A (наш, коммерческий транспорт):**

| Подсегмент | Ключевые критерии | Приоритет |
|------------|-------------------|-----------|
| **A1** | Нужно срочно + пишут сами (инициатива клиента); свой автопарк; **более 5 авто** | Высший. Быстрая реакция, приоритет доставки. |
| **A2** | Свой автопарк; менее 5 авто; или срочность ниже / меньше инициативы со стороны клиента | Высокий. Стандартный офер, чек-листы процесса. |

Метрики для A1 (число машин, размер автопарка, ежемесячный пробег, частота покупок) — источник: **БД по company_id** (amoCRM компании, <internal-component>, выгрузки). См. проект [<client> A1 metrics from DB](mdc:[<internal-component>]/clients/all-clients/m-<client>-ru/<client>-a1-metrics-db/<client>-a1-metrics-db.todo.md).

**Действия по сегментам в amoCRM:**

- **Теги для квалификации:** `иномарки`, `коммерческий транспорт`, `импорт`, `автопарк`, `A1`, `A2` — для фильтрации по типу CTO, транспорта и приоритету.
- **Не наш сегмент (CTO с иномарками):** не перегружать процесс, не гнаться за доставкой «несколько раз в день»; при анти-триггерах («нужна доставка 2–3 раза в день») относить к D.
- **Наш сегмент (CTO коммерческий транспорт):** приоритет A/B; A1 — максимальный приоритет (срочность, автопарк 5+); A2 — высокий; усиливать офер по «след. день» и «день в день» там, где мы реально можем.

### Блокеры: симптом → гипотеза → как работать (формат для команды продаж)

По [Sales Marketing Product CJM Standard](abstract://standard:sales_marketing_product_cjm_standard): блокеры вести в формате таблицы (симптом | гипотеза | влияние | действия). Примеры из практики advising (выгружены из сессии с селзом и дочитаны по деталям выгрузки):

| Симптом | Гипотеза | Влияние | Как работать |
|--------|----------|---------|--------------|
| Не берёт трубку | Звоним не вовремя; нет альтернативного канала | Риск потери лида | Договориться о времени (письмом/мессенджер); дублировать инфо в мессенджер и на почту. |
| Не отвечает в мессенджер | Контакт устарел или не смотрит; мы не дублируем на почту | Сделка зависает | Дублировать на email; выставить «Дата след. контакта» и задачу на повтор. |
| Бросил трубку (напр. «у врача») | Звоним в неподходящий момент | Разовый срыв, при повторении — анти-триггер | Сначала сообщение: «Когда удобно перезвонить?»; фиксировать «удобное время». |
| «Не берет» без пояснения | Причина не зафиксирована | Блокер: непонятно, теряем навсегда или «пока не актуально» | Развести в теги/этапы; дублировать каналом; в примечании: «отправили X, ждём до даты Y». |
| Подэтап «Компания ликвидируется» | Объективный выход клиента с рынка | Потеря клиента | Отдельный этап/тег; архивировать. |
| «У нас дороже» (подэтап/тег) | Ценность/офер не закрыты (сегмент D) | Анти-триггер, возможен уход | Фиксировать; анализировать позиции; запланировать контакт через N дней (офер/новость). |
| Пустая или просроченная «Дата след. контакта» при «Договор подписан» | Нет следующего шага; общение не возобновлено | Риск потери | Заполнять дату при любом действии; выгружать просроченные и проходить с менеджерами; в примечании: «след. шаг: X, дата: Y». |

Таблицу обновлять по мере верификации на звонках и из примечаний; использовать как чеклист при разборе сделок с командой продаж.

### Алгоритм сегментации по прохождению по сделке (кратко)

1. **Выгрузить сделки из CRM, Customer Relationship Management**: этап, теги, примечания, дата след. контакта, при необходимости джойн с компаниями (ИНН, контекст).
2. **Фаза 1:** Сессия с селзом ~20 сделок → выписать симптомы и корневые блокеры (теги, этапы, типичные фразы).
3. **Фаза 2:** По выгрузке искать триггеры/анти-триггеры по спискам выше; отмечать застревания (этап CJM, геп-фразы, дата след. контакта).
4. **Квалифицирующий вопрос:** По каждому застреванию — «про нас» или «про клиента» → B/X или D/C.
5. **Маппинг:** этап + тег + фразы + дата след. контакта → A/B/C/D/X; зафиксировать таблицу «симптом → гипотеза → как работать».
6. **Чеклист телс-сигналов:** Зафиксировать финальные правила для разметки следующих выгрузок (и при необходимости колонку `abcdx_segment` или отчёт по сегментам).

## 🧠 Awareness · Behavior · Context · Desire (другая модель — оси сегментации)

**Не путать с ABCDX.** Это модель по четырём осям: осведомлённость, поведение, контекст, желание.

### Awareness · Behavior · Context · Desire Framework

**A (Awareness) — Осведомлённость:**
- Клиенты знают о проблеме
- Понимают необходимость решения
- Активно ищут решения

**B (Behavior) — Поведение:**
- Клиенты демонстрируют определённое поведение
- Показывают паттерны использования
- Имеют характерные метрики

**C (Context) — Контекст:**
- Клиенты находятся в определённом контексте
- Имеют специфические условия работы
- Обладают уникальными характеристиками

**D (Desire) — Желание:**
- Клиенты хотят достичь определённых целей
- Имеют чёткие желания и мотивации
- Показывают готовность к действию

### Интеграция с JTBD

Каждый сегмент (по любой из двух моделей) при необходимости дополняется:
- **JTBD-сценарий** — описание задачи клиента
- **Struggling moment** — момент борьбы
- **Desired outcome** — желаемый результат
- **Success metric** — метрика успеха

## 🔄 Шаг 1 — Анализ клиентской базы (SGR: Cascade)

### Назначение шага

Преобразовать неструктурированную клиентскую базу в структурированные данные для сегментации.

### Схема (SGR Cascade Pattern)

```python
from pydantic import BaseModel
from typing import Annotated
from annotated_types import MinLen, MaxLen

class ClientBaseAnalysis(BaseModel):
    """Схема анализа клиентской базы (SGR Cascade)"""
    raw_client_data: Annotated[str, MinLen(10), MaxLen(5000)]
    structured_data: Annotated[Dict, MinLen(1)]
    identified_patterns: Annotated[List[str], MinLen(1)]
    missing_data: Annotated[List[str], MinLen(0)]
```

### Правила заполнения

- `raw_client_data` — исходные данные клиентов
- `structured_data` — структурированные данные для анализа
- `identified_patterns` — выявленные паттерны поведения
- `missing_data` — недостающие данные для сегментации

### Output-чеклист

- [ ] Данные структурированы и готовы к анализу
- [ ] Выявлены паттерны поведения клиентов
- [ ] Определены недостающие данные
- [ ] Все поля заполнены конкретными данными

## 🔄 Feedback Loop

### Протокол обратной связи

1. **Сбор данных об использовании:**
   - Отслеживание конверсии по сегментам
   - Анализ эффективности оферов
   - Сбор обратной связи от клиентов

2. **Анализ результатов:**
   - Сравнение метрик до и после сегментации
   - Выявление проблемных сегментов
   - Анализ причин низкой эффективности

3. **Итеративное улучшение:**
   - Обновление критериев сегментации
   - Уточнение JTBD-сценариев
   - Улучшение оферов

4. **Документирование изменений:**
   - Фиксация изменений в сегментации
   - Обновление стандарта
   - Обмен знаниями с командой

### Метрики обратной связи

- **Частота обновления сегментации:** ежемесячно
- **Порог для пересмотра:** изменение конверсии > 10%
- **Критерий успеха:** улучшение метрик на > 15%

## 📚 Библиотека примеров

### Пример 1: B2B SaaS сегментация

**Сегмент A1B1C1D1 (High Awareness, Active Behavior, Startup Context, High Desire):**
- **Характеристики:** Знают о проблеме, активно используют продукт, стартап, высокое желание масштабироваться
- **JTBD:** "Когда я хочу масштабировать стартап, я хочу получить структурированный план роста"
- **Офер:** "Помогаем стартапам создать структурированный план масштабирования с четкими метриками и этапами"

### Пример 2: Enterprise сегментация

**Сегмент A2B2C2D2 (Medium Awareness, Passive Behavior, Enterprise Context, Medium Desire):**
- **Характеристики:** Частично знают о проблеме, используют продукт редко, enterprise, среднее желание улучшений
- **JTBD:** "Когда я работаю в enterprise, я хочу улучшить процессы без риска"
- **Офер:** "Помогаем enterprise компаниям безопасно улучшить процессы через пошаговые изменения"

## ✅ Output и доставка

### Output для обновления стандарта
- **Формат**: `.md` файл в папке `<standard-ref>
- **Название**: `1.0 abcd client segmentation standard [date time] cet by [author].md`
- **Канал**: Git repository
- **Подтверждение**: ожидается от пользователя

### Output для использования стандарта
- **Формат**: Применение ABCDX и/или Awareness · Behavior · Context · Desire для сегментации клиентов
- **Структура**: Следование шагам методологии из раздела "Шаги сегментации"
- **Канал**: Git repository, база знаний advising
- **Подтверждение**: валидация через чеклисты input/output/outcome
