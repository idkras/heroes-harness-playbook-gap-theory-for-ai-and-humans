## 🎯 Цель документа

Создать стандарт для Telegram-агента чемпионов (champions), который внедряет концепции Gap Theory через ведение направляющими вопросами в стиле Socratic Thinking Processes. Цель — распространять изменения путём снижения Glue Effort (Effort₂) и преобразования ландшафта усилий. Агент учит, не читая лекций: задаёт вопросы, предлагает эксперименты, сопоставляет истории пользователей с терминами Gap Theory. Стандарт включает методы измерения по переписке в чатах и логам задач проектов (GDEM — Gap Dynamics Energy Model).

## 0. Purpose (Назначение)

1. Научить чемпионов видеть топологию изменений в компании.
2. Выявлять аттракторы, стоки (sinks) и узкие места по сообщениям, задачам и логам проектов.
3. Выбирать действия малые, обратимые и эффективные.
4. Снижать Glue Effort и пошагово менять ландшафт усилий.
5. Измерять энергию системы на реактивное поведение vs проактивное изменение через GDEM.

## 1. Champion Role (Роль чемпиона)

1. Чемпион — навигатор ландшафта усилий.
2. Чемпион выявляет гепы и зоны высокого Glue Effort.
3. Чемпион запускает малые обратимые изменения, снижающие Effort₂.
4. Чемпион распространяет маршруты, а не идеи.

## 2. Core Concepts to Teach (Ключевые концепции)

1. **Flow** — валидированная доставка, не активность.
2. **Stock** — накопленные ограничения и риски.
3. **Effort₁** — усилие исполнения (make effort).
4. **Effort₂ (Glue Effort)** — усилие связности и координации.
5. **Gap types:**
   - Semantic (смысл)
   - Interface (контракт/формат)
   - Temporal (тайминг/очереди)
   - Authority (кто решает/владеет риском)
   - Observability (видимость последствий)
6. **Landscape** — геометрия дешёвых и дорогих маршрутов.
7. **Attractor** — устойчивый паттерн, куда задачи и энергия «падают» сами.
8. **Gap Sink** — этап, куда задачи приходят и останавливаются.
9. **Capacity of Relaxation** — максимальный поток без роста токсичного Stock.
10. **Human to Agent capacity limit:** Nmax = T / Effort₂ per agent.

## 3. Quick Scan from Chats and Task Systems

### 2.1 Incident Wave (метрики из логов)

**Метрики:**
- New tasks per week
- Closed tasks per week
- Active tasks count over time
- **Incident Half Life** — время, за которое active count падает вдвое
- **Damping Ratio** = New per week / Closed per week

**Интерпретация:**
- Damping Ratio > 1 → система «греется»
- Half Life растёт → система теряет релаксацию

**Шаги чемпиона:**
1. Прекратить добавлять scope.
2. Найти одно место с максимальными сигналами Effort₂ (много сообщений, уточнений, апрувалок).
3. Снизить один малый кусок Glue Effort там.

### 2.2 Cumulative Flow with Waiting Zones

**Колонки на доске:**
- New
- In progress
- Waiting for decision
- Waiting for clarification
- Waiting for approval
- Waiting for data
- Done

**Метрики:**
- Ширина waiting zones (время в этих колонках)
- Скорость роста In progress vs Done

**Интерпретация:**
- Широкие waiting zones = высокий Glue Effort
- In progress растёт быстрее Done → токсичный Stock растёт

**Шаги чемпиона:**
1. Выбрать самую широкую waiting zone.
2. Классифицировать доминирующий тип гепа.
3. Убрать одну апрувалку или добавить один инвариант или одну автоматическую проверку.

## 4. Detecting Attractors and Sinks

### 3.1 Признаки

1. Задачи приходят на один и тот же этап и останавливаются.
2. Много обсуждений, мало валидированной доставки.
3. Одни и те же люди становятся узкими местами.
4. Одни и те же проблемы повторяются под разными именами.

**Метрики из логов:**
- **Gap Sink Index** — доля задач, застрявших в одном этапе дольше T дней
- **Stage Entropy** — неравномерность распределения времени по этапам
- **Node Load** — число решений и хэндоффов через одного человека/команду

**Интерпретация:**
- Высокий Gap Sink Index = устойчивый sink
- Низкая Stage Entropy с одним доминирующим этапом = аттрактор
- Очень высокий Node Load = топологическое узкое место

**Шаги чемпиона:**
1. Не атаковать аттрактор напрямую.
2. Найти обход, снижающий Effort₂ для одного класса задач.
3. Сделать обход видимым и безопасным.
4. Дать другим скопировать маршрут.

## 5. Project Death Zones

**Метрики:**
- **Decision to Delivery Ratio** — время от первого решения до первой доставки
- **Commitment Decay Curve** — активные стейкхолдеры во времени
- **Effort₂ Accumulation Rate** — рост сообщений, апрувалок и уточнений на единицу доставки

**Интерпретация:**
- Decision to Delivery растёт → система теряет throughput
- Стейкхолдеры падают рано → проект мёртв до отмены
- Effort₂ растёт быстрее Flow → проект не выживет

**Шаги чемпиона:**
1. Сжать scope до одной доставляемой части.
2. Убрать одну зависимость или одну апрувалку.
3. Добавить один rollback или один safe flag.
4. Цель — первая доставка, не полное решение.

## 6. Estimating Task Complexity by Topology

**Метрики:**
- **Dependency Degree** — число затронутых команд/систем
- **Gap Count** — число типов гепов в цепочке
- **Cycle Count** в зависимостях

**Правила:**
- Два гепа на критическом пути ≈ ×10 замедление
- Четыре и более коррелированных гепа ≈ коллапс throughput
- Каждый цикл умножает Effort₂

**Шаги чемпиона:**
1. При высоком Dependency Degree — разбить задачу.
2. При высоком Gap Count — снизить порядком или контрактом.
3. При цикле — разорвать: чёткий owner или один шаг асинхронно.

---

## 7. Connectivity Graph and Viscosity

**Метрики из логов и org graph:**
- **Betweenness Centrality** — число кросс-доменных хэндоффов на узел
- **Average Path Length** — число переходов статуса от идеи до доставки
- **Node Load concentration**

**Интерпретация:**
- Один–два экстремально центральных узла = хрупкая топология
- Растущий path length = кристаллизация структуры
- Высокая концентрация = скрытый authority или семантические гепы

**Шаги чемпиона:**
1. Добавить один общий артефакт/инвариант, снижающий нагрузку перевода.
2. Переместить одно решение ближе к исполнению.
3. Создать один прямой маршрут в обход комитета/релея.

---

## 8. System Persistence

**Метрики:**
- **Project Survival Curve** — доля проектов, дошедших до первой и второй доставки
- **Scope Shrink Ratio** — planned scope / delivered scope
- **Reactivation Rate** — доля отменённых проектов, которые возвращаются

**Интерпретация:**
- Мало проектов до второй итерации → система не выдерживает изменения
- Большое scope shrink → система не переваривает сложность
- Низкая reactivation → система теряет обучение

**Шаги чемпиона:**
1. Начинать с проектов, укладывающихся в текущую capacity.
2. Проектировать изменения обратимыми.
3. Вести простой реестр уроков и переиспользовать его.

## 9. Hidden Axes

### 8.1 Semantic Drift

**Метрики:** количество переформулировок той же цели; споры об acceptance criteria.

**Действия:** один shared definition of done; один пример корректного outcome.

### 8.2 Trust Bandwidth

**Метрики:** доля изменений, проходящих без тяжёлого ревью.

**Действия:** trigger-based review; одна дешёвая автоматическая проверка.

### 8.3 Reversibility Index

**Метрики:** время до rollback или отключения.

**Действия:** feature flags; canary или shadow mode.

### 8.4 Learning Half Life

**Метрики:** время до повторения той же ошибки.

**Действия:** одна короткая post-incident заметка; превратить в инвариант или проверку.

## 10. States of the System and Actions

### State A: Heating

**Признаки:** Damping Ratio > 1; waiting zones растут; Effort₂ доминирует.

**Шаги:** прекратить scope; выбрать одну waiting zone; убрать один геп; измерить за неделю.

### State B: Stagnation

**Признаки:** задачи двигаются, но ничего не ship'ится; одни и те же обсуждения; одни и те же bottlenecks.

**Шаги:** выявить главный gap sink; создать один bypass; сделать его безопасным и видимым; дать одной команде попробовать.

### State C: Over-control

**Признаки:** слишком много апрувалок; страх ship'ить; беклог ревью растёт.

**Шаги:** классифицировать уровни риска; trigger-based review; rollback или мониторинг; убрать один шаг апрувала.

### State D: Fragmentation

**Признаки:** много хэндоффов; высокая нагрузка перевода; конфликтующие определения.

**Шаги:** один shared артефакт/контракт; закрыть один семантический геп; укоротить один путь в графе.

### State E: Healthy but Slow

**Признаки:** Flow есть, но низкий; Stock стабилен; Effort₂ всё ещё высок.

**Шаги:** оптимизировать один частый путь; автоматизировать одну проверку; убрать одно повторяющееся уточнение.

## 13. Agent Dialogue Protocol

### Input

Пользователь описывает проблему или трение.

### Step 1: Role and Risk

- Спросить: кто пользователь в этой ситуации.
- Спросить: какой риск неприемлем для этой роли.

### Step 2: Symptom Gap

- Спросить: что болит прямо сейчас.
- Спросить: как пользователь знает, что болит.

### Step 3: Gap Classification

Спросить вопросы для классификации гепа:

- Про смысл (meaning)?
- Про контракт или формат?
- Про тайминг или очереди?
- Про то, кто решает или владеет риском?
- Про видимость последствий?

### Step 4: Glue Effort Estimation

- Спросить: где горят время и внимание.
- Спросить: где появляются переделки или уточнения.

### Step 5: Small Morphism Proposal

- Предложить одно обратимое изменение.
- Проверить: снижает ли оно Effort₂.
- Проверить: соответствует ли риску роли.

### Step 6: Measurement

- Спросить: как увидеть улучшение за неделю.

### Step 7: Next Step

- Если улучшение видно — предложить следующий малый шаг.
- Если не видно — пересмотреть гипотезу гепа.

4. Агент постепенно строит общий словарь с пользователем.

## 15. Virus of Change Mechanism

### Rule

Изменения распространяются, когда маршрут становится дешевле.

### Local Loop

1. Найти одну точку высокого Effort₂.
2. Применить один малый морфизм.
3. Показать, что стало легче.
4. Соседи копируют маршрут.

### Scaling

1. Агент собирает успешные паттерны.
2. Агент предлагает их другим чемпионам в похожих контекстах.

## 16. Protocol Without Gaps (Протокол без гепов)

1. **Plan:** Агент формулирует шаги рассуждения.
2. **Gap check:** Агент перечисляет возможные точки путаницы.
3. **Explanation:** Агент следует плану по шагам; определяет каждый термин перед использованием.
4. **Verification:** Агент спрашивает, можно ли восстановить цепочку без догадки.

---

## 17. Protocol of Coherence (Протокол связности)

1. **Descending loop:** От проблемы пользователя найти корневой разрыв.
2. **Bridge:** Ввести минимальные определения и инварианты.
3. **Ascending loop:** Собрать решение и обобщить.
4. **Metrics:**
   - Два гепа в критической цепочке → throughput падает примерно в 10 раз.
   - Четыре и более коррелированных гепа → throughput коллапсирует.

## 22. Daily Champion Routine

1. Смотреть waiting zones и sinks.
2. Спрашивать, какой геп там доминирует.
3. Предлагать одно малое обратимое изменение.

## 25. GDEM — Gap Dynamics Energy Model

Цель: считать «энергию» системы на реактивное (латание симптомов) vs проактивное (устранение корней) поведение.

### 25.1 Объекты и пространства

- **S(t)** = (G, X(t), U(t), R(t), K(t))
- **G** = (V, E) — граф людей, ролей, ботов, процессов
- **X(t)** — состояние узлов (очереди, контекст, решения, артефакты)
- **U(t)** — активные воздействия (задачи, решения, коммуникации)
- **R(t)** — ресурсы (время, внимание, деньги, compute)
- **K(t)** — ограничения (SLA, регуляторика, политика, зависимости)

### 25.2 Гепы

- **g_k(t)** = требование_k(t) − способность_k(t)
- **g^S** (симптомные) — баги, инциденты, жалобы
- **g^R** (корневые) — архитектура, процесс, стимулы, модель данных
- **g^C** (связностные) — потеря контекста, разрыв сигнала

**Операционально:** симптом — снижает g^S быстро, g^R не меняется; корень — снижает g^R и будущие g^S.

### 25.3 Метрики гепов

| Метрика | Формула |
|---------|---------|
| Амплитуда | A_k(t) = ‖g_k(t)‖ |
| Напряжение | Φ_k(t) = A_k(t) · W_k(t) |
| Скорость нарастания | ḡ_k(t) = dA_k/dt |
| Риск каскада | Cascade_k(t) = Σ_j P(k→j) · Φ_j(t) |

### 25.4 Энергия: реактивная vs проактивная

**Локальная энергия узла i:**
```
E_i(T) = ∫₀ᵀ [ c_time·τ_i(t) + c_ctx·κ_i(t) + c_coord·χ_i(t) ] dt
```
где τ_i = активное время, κ_i = потери контекста, χ_i = координация (сообщения, созвоны, апрувалки).

**Разделение:** type(a) ∈ {reactive, proactive, mixed}
- E_i^R(T) = ∫ cost_i · I[reactive] dt
- E_i^P(T) = ∫ cost_i · I[proactive] dt
- **ρ(T)** = E^R / (E^R + E^P) — цель: ρ падает

### 25.5 Классификация действий из логов

**Reactive-score(a)** = σ(incident_link, urgency, rollback/patch, unplanned, context_switch, customer_escalation − architecture_change, instrumentation, automation)

**Proactive-score(a)** = σ(root_cause_link, prevention, reduces_future_incidents, improves_observability, reduces_coordination_cost, reduces_lead_time)

### 25.6 Инварианты

1. **I_stab** = Σ Φ_k — не должен расти
2. **ρ** — должен падать
3. **C** (cohesion) — должен расти
4. **O** (observability) — доля действий с улучшением измерения
5. **K_change** — медианная стоимость изменения — должен падать

### 25.7 Мощность и КПД

- **Power_i** = dW_i/dt, где W_i — вклад в снижение Φ_k
- **η_i** = W_i / E_i — низкий η = энергия в трение

### 25.8 Lever Index (качество инъекции)

```
Lever(I) = (Δḡ_root + ΔCascade) / (Energy_cost(I) + Coordination_cost(I))
```

**Симптомная ловушка:** инциденты падают краткосрочно, но ρ и Cascade не падают → инъекция симптомная.

### 25.9 Коммутаторы (порядок операций)

**[a, b]** = a∘b − b∘a  
- [a,b] = 0 → независимость
- ‖[a,b]‖ большое → порядок критичен

**Типовые пары:** observability затем speed безопаснее; schema затем automation безопаснее; ownership затем SLA безопаснее.

### 25.10 Импульс и направление

- **G⃗(t)** = (A_1(t), ..., A_n(t)) — вектор гепов
- **M_L** = f(размер, связность, регуляторика, стейкхолдеры) — масса/инерция
- **P⃗_L(t)** = M_L · dG⃗/dt — импульс; если направлен в уменьшение корневых гепов — система проактивна
- **D(t)** = proj_root(P⃗) / (|proj_root| + |proj_sym|) — индекс направления; цель D → 1

### 25.11 Мощность и тепло на уровне L

- **Power_L** = Σ Power_i − overhead
- **Heat_L** = E_L^R / (E_L^R + E_L^P) — то же ρ на уровне L

### 25.12 Эластичность инъекции

**Elasticity(I)** = (Δρ/ρ) / (ΔE_total/E_total) — сколько реактивности исчезло на единицу затраченной энергии.

### 25.13 Локальная модель узла

- **capacity_i**, **latency_i**, **noise_i** — пропускная способность, задержка решения, искажение смысла
- **Flow_i**, **Backlog_i**, **WIP_i** — задачи/время, ожидающие, в работе
- **Friction_i** = c_ctx·κ_i + c_coord·χ_i + c_switch·switches_i
- **Cascade_contrib_i** = Σ_{tasks touching i} ΔCascade_k

### 25.14 Физические аналогии

- **H_queue** — энтропия очередей; рост = потеря приоритизации
- **T** (температура) — дисперсия сроков/результатов; высокая T = шум, реактивность
- **D** (диссипация) — энергия в трение; цель — снижать
- **Фазовые переходы:** порог ρ → система живёт пожарами; порог O → самообъясняемость
- **Tension field** — поле напряжения по задачам/командам; инъекция меняет ландшафт, не впрыскивает силу

### 25.15 Дашборды

1. ρ(t) по уровням (команда/отдел/компания)
2. Φ_total(t), Cascade_total(t)
3. Power_i, η_i
4. Heat map графа по E_i^R и Cascade_contrib_i
5. Матрица коммутаторов ‖[a,b]‖ по типам инициатив

### 25.16 Практический протокол внедрения GDEM

1. Таксономия задач: reactive/proactive + root/symptom
2. Атрибутировать эффект по Φ_k и Cascade_k
3. Считать E^R, E^P, ρ, Power, η, Lever, Elasticity
4. **Правило:** каждый спринт минимум X% энергии в корни; ускорение релизов только после наблюдаемости
5. **Еженедельный отчёт:** где выросла реактивность; какие корневые гепы снизились; какие инъекции дали max Lever

### 25.17 Расширения (Extensions)

- Оценка массы M через граф зависимостей и стейкхолдеров
- Базис ê_root, ê_sym через причинные графы P(k→j)
- Классификатор reactive/proactive по тексту задач
- Связь с метрикой dt-связности
- Поиск аттракторов: устойчивые зоны высокого Φ и высокой диссипации

## 26. Minimal Data: Как считать из чатов и задач

### 26.1 Логи задач (Jira/Linear/Notion/Slack→tasks)

| Поле | Назначение |
|------|------------|
| created_at, started_at, done_at | Half Life, Damping Ratio, path length |
| labels: incident, bug, feature, refactor, observability, automation | Reactive/Proactive классификация |
| links: root_cause, incident_id | Cascade, symptom vs root |
| assignee, team, dependencies | Node Load, Betweenness, Dependency Degree |
| story_points / effort | E_i, τ_i |

**Формулы:**
- **Damping Ratio** = COUNT(created WHERE week=W) / COUNT(done WHERE week=W)
- **Half Life** = время, за которое active_count падает в 2 раза (регрессия по дням)
- **Gap Sink Index** = COUNT(tasks WHERE stage=X AND days_in_stage > T) / COUNT(all)
- **Node Load** = COUNT(handoffs WHERE node=i) + COUNT(decisions WHERE node=i)

### 26.2 Логи коммуникаций (Slack/Telegram)

| Поле | Назначение |
|------|------------|
| message_count per task | χ_i (coordination cost) |
| participants, thread_duration | coupling, κ_i |
| time_to_first_response | latency_i |
| context_switches per person per day | c_switch · switches_i |

**Формулы:**
- **χ_i(t)** ∝ message_count(t) + approval_count(t) + meeting_hours(t)
- **κ_i(t)** ∝ context_switches(t) · avg_restore_time

### 26.3 Логи инцидентов

| Поле | Назначение |
|------|------------|
| severity, duration, affected_users | Φ_k, W_k |
| repeat_rate | Cascade, P(k→j) |
| time_to_recovery | реактивность |

### 26.4 Маппинг: чат → метрика

| Сигнал в чате | Метрика GDEM |
|----------------|--------------|
| «Опять то же самое» | High Gap Sink Index, low Stage Entropy |
| «Ждём ответа от X» | Node Load, waiting zone width |
| «Нужно согласовать с 5 людьми» | χ_i, K_change, Dependency Degree |
| «Не понял, что имели в виду» | κ_i, Semantic Drift |
| «Тушим пожар» | Reactive-score, ρ |
| «Добавили алерты» | Proactive-score, O |

## 27. Champion Checklist with GDEM

1. Прочитать waiting zones и sinks.
2. Измерить ρ, Half Life, Damping Ratio.
3. Выбрать одну инъекцию с высоким Lever и низкой Coordination cost.
4. Проверить коммутаторы с текущими инициативами.
5. Запустить эксперимент на неделю.
6. Сохранить, если Effort₂ и ρ снижаются; иначе откатить.
