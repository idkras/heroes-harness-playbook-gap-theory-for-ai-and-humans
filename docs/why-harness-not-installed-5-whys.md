# 5 Whys — почему по умолчанию харнесс целиком не устанавливался

**Дата:** 2026-06-19
**Bead:** mike-harness-template-uy8
**Симптом:** после `git clone` SessionStart-хук печатает `🟢 harness готов`, но `bd`,
Dolt, python-зависимости и graphify **не установлены** — их приходится ставить руками.
В соседней сессии («Folder sync setup») это вскрылось как `bd not found`, нет `.beads/`.

## Цепочка «5 почему»

1. **Почему toolchain не установился сам?**
   Потому что единственный SessionStart-хук — `scripts/harness_bootstrap.py` — по
   замыслу ставил только *обвязку* (chmod хуков, портабилизация путей, checksum,
   wiring), а инструменты (`bd`/Dolt/deps) — нет.

2. **Почему bootstrap не ставил инструменты?**
   Потому что bootstrap намеренно **stdlib-only** («Ничего внешнего»), а установка —
   это сетевые операции (brew/curl/pip). Их сознательно вынесли из bootstrap…

3. **…вынесли — куда?**
   В `scripts/setup/install_beads.sh` — но этот установщик **не входит в публичный
   шаблон**: он лежал на canonical-машине владельца и в git-экспорт не попал.
   В шаблоне остались только *упоминания* в скилах (легенда без артефакта).

4. **Почему установщик не попал в публичный экспорт?**
   Потому что экспорт собирался по принципу «leak-free»: всё, что пахло локальными
   путями/секретами/canonical-инфраструктурой, отрезалось. Установщик отрезали
   вместе с canonical-слоем — и **не заменили публичным аналогом**.

5. **Почему отсутствие установщика никто не поймал автоматически?** *(корневая)*
   Потому что **не было проверки-гейта на наличие самого установщика и на полноту
   toolchain**. `harness_guardian_check.py` проверял *результат* graphify
   (`graphify-out/graph.json`) как WARN, но никакой шаг не проверял «а установлен ли
   вообще bd/Dolt/deps и есть ли чем их ставить». Разрыв (gap) между ожиданием
   «harness готов» и реальностью «инструментов нет» был **невидим для самого харнесса**.

## Корневая причина (root cause)

> Публичный шаблон самоактивировал **обвязку**, но не **инструментарий**, и не имел
> ни (а) публичного установщика toolchain, ни (б) гейта, который бы заметил его
> отсутствие. Классический gap «execution + feedback»: шаг исполнения вырезан,
> и петля обратной связи о его отсутствии тоже отсутствовала.

## Что чиним (этот PR)

| Слой gap | Фикс |
|---|---|
| execution (нет установщика) | `scripts/setup/install_all.sh` — публичный idempotent-установщик (venv+deps, bd, Dolt, bd init; graphify помечен optional/canonical-only) |
| execution (не запускался сам) | `harness_bootstrap.py` → шаг `toolchain`: на первом SessionStart один раз (marker-guarded) зовёт установщик |
| feedback (разрыв невидим) | тот же шаг `toolchain` теперь в getting-started чеклисте: PASS / WARN с конкретной командой |
| knowledge (Миша не знал как) | `docs/GETTING_STARTED.md` — структура, скилы, агенты, как ставить и вести задачи |

## Фальсификация фикса

**Claim:** «теперь после клона всё ставится само».
**Проверка:** `bash scripts/setup/install_all.sh --check` → exit≠0 при неполноте;
`python3 scripts/harness_bootstrap.py` на чистом клоне → шаг `toolchain` PASS после
авто-установки. graphify остаётся WARN осознанно (нет публичного источника) — это
**честный** предел, а не скрытый разрыв.
